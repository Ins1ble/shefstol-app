# Test Plan · Shefstol Web App (MVP)

## 1. Идентификатор плана
**MTP-SHEF-MVP-1.0**

## 2. Введение
Настоящий документ представляет собой Мастер-план тестирования клиентского веб-приложения **«Шеф‑стол»** (далее — Продукт).  
Продукт создан для самостоятельной записи клиентов на кулинарные мастер-классы и полностью заменяет ручной процесс согласования через WhatsApp и Google-таблицы.

Документ определяет стратегию, методы, уровни, ресурсы и график тестирования, а также устанавливает критерии качества и завершения тестирования.  
Он является руководящим документом для всей команды QA на время подготовки MVP к запуску.

Все артефакты тестирования размещаются в директории `03-test/`.

## 3. Объект тестирования
### 3.1 Продукт
Одностраничное веб-приложение (SPA), оптимизированное для мобильных устройств, с REST API на Flask + SQLite.

### 3.2 Компоненты
- **Backend:** REST API, реализующее полный цикл работы клиента – от авторизации до отмены брони.
- **Frontend:** HTML5/CSS3/JavaScript (Bootstrap 5), экраны SCR‑001…SCR‑007 и модальные окна BS‑001…BS‑004.
- **База данных:** SQLite, WAL-режим, seed-данные (2 слота, 3 шефа, 2 программы).
- **Интеграции:** внешний сервис статических карт Яндекс (static‑maps), Google Календарь, системные уведомления браузера (Notification API).

### 3.3 Ключевые бизнес-цели (трассировка на метрики успеха)
- **BR‑2:** ≥70% записей совершаются онлайн самостоятельно → проверяется через пользовательские сценарии.
- **BR‑1:** 0 двойных броней / конфликтов → валидация атомарности бронирования.
- **BR‑5:** снижение простоев из‑за поздних отмен → проверка правила 2‑часовой отмены.
- **BR‑7:** запуск MVP в срок ~2 месяца → подтверждается готовностью к приёмочному тестированию.
- **BR‑8:** оплата офлайн, приложение фиксирует запись → проверка отображения цены и отсутствия платёжных элементов.

## 4. Цели тестирования
1. Подтвердить, что Продукт реализует **все функциональные (FR) и нефункциональные (NFR) требования**, зафиксированные в аналитике (`01-analysis/2-requirements/`).
2. Проверить **корректность бизнес-логики**: правило 2‑часовой отмены, раздельный учёт мест и проката, идемпотентность бронирования.
3. Гарантировать **обработку граничных и ошибочных сценариев** согласно матрице ошибок R‑023 и use‑cases (UC‑1, UC‑2, UC‑3).
4. Оценить **удобство использования на мобильных устройствах** (mobile‑first, крупные touch‑зоны, контрастность, адаптивность).
5. Удостовериться в **безопасности**: доступ только к своим данным, корректная работа refresh‑токенов, невозможность чужих операций.
6. Оценить **производительность** API и отзывчивость UI при пиковых нагрузках (до 50 одновременных пользователей).
7. Обеспечить **отсутствие критических дефектов** перед приёмочным тестированием.

## 6. Стратегия тестирования
### 6.1 Подход
Комбинация ручного и автоматизированного тестирования.  
Ручное тестирование применяется для проверки UI/UX, приёмочного тестирования и исследовательского тестирования.  
Автоматизированное тестирование покрывает API (юнит/интеграционное) и критические E2E-сценарии.

### 6.2 Уровни тестирования
| Уровень | Описание | Инструменты | Ответственный |
|---------|----------|-------------|---------------|
| **Модульное (unit)** | Изолированная проверка функций API (эндпоинты, обработчики ошибок, бизнес-правила) | pytest | QA |
| **Интеграционное** | Проверка взаимодействия между модулями API (авторизация + бронирование, работа с БД) | pytest | QA |
| **Системное (E2E)** | Полные пользовательские сценарии через UI браузера | Ручное + Selenium | QA |
| **Приёмочное** | Соответствие критериям приёмки из SCR/BS‑документов | Ручное | QA + Заказчик |

### 6.3 Типы тестирования
- **Функциональное** – проверка всех экранов и операций.
- **UX/UI** – мобильная адаптация, размеры touch‑зон, контрастность, динамический шрифт.
- **Безопасность** – контроль доступа (401/403), валидация токенов, удаление персональных данных.
- **Производительность** – время отклика API (p95 < 2.5 с загрузка слотов, <1.5 с create/cancelBooking), нагрузочное тестирование (k6, опционально).
- **Регрессионное** – автоматический прогон юнит/интеграционных тестов после каждого изменения кода.

### 6.4 Приоритизация тест‑кейсов
- **P0 (Critical)** – основные сценарии (вход, просмотр слотов, бронирование, отмена). Без них продукт не запускается.
- **P1 (High)** – важные функции (фильтрация, профиль, удаление аккаунта, уведомления).
- **P2 (Medium)** – второстепенные (отображение карты, кнопка «В календарь»).

### 6.5 Матрица трассировки требований → тесты
Для каждого FR и US будет разработан минимум один тест‑кейс (см. `test-cases.md`).  
Пример трассировки:

| Требование | Тест‑кейс(ы) | Приоритет |
|------------|--------------|-----------|
| FR‑1, FR‑2, FR‑43 | TC‑AUTH‑001…006 | P0 |
| FR‑9, FR‑38 | TC‑SLOTS‑001…005 | P0 |
| FR‑10…FR‑15, FR‑45, FR‑50 | TC‑BOOK‑001…015 | P0 |
| FR‑16…FR‑18, FR‑46 | TC‑CANCEL‑001…006 | P0 |
| FR‑35a | TC‑MYBOOK‑001…003 | P1 |
| FR‑47…FR‑49 | TC‑PROF‑001…004 | P1 |
| FR‑33, NFR‑13 | TC‑PUSH‑001 | P1 |
| NFR‑1, NFR‑2, NFR‑6 | TC‑UX‑001…003 | P1 |
| NFR‑8, NFR‑9 | TC‑CON‑001…003 | P0 |

## 7. Тестовое окружение
### 7.1 Окружение Backend
- **ОС:** Windows 10/11, macOS, Linux
- **Рантайм:** Python 3.13
- **Веб-сервер:** Flask development server (`app.run(debug=True, port=5000)`)
- **База данных:** SQLite 3, WAL‑режим, файл `shefstol.db`
- **Тестовые данные:** 2 слота, 3 шефа, 2 программы, заполняются автоматически при старте.

### 7.2 Окружение Frontend
- **Браузер:** Google Chrome последней версии (основной)
- **Дополнительно:** Firefox, Safari (для проверки кросс‑браузерности, опционально)
- **Эмуляция мобильных устройств:** Chrome DevTools (iPhone 12 Pro, Pixel 5)

### 7.3 Инструменты
| Инструмент | Назначение |
|------------|------------|
| **pytest** | Юнит и интеграционное тестирование API |
| **Selenium WebDriver** | E2E‑тесты в Chrome |
| **Chrome DevTools** | Ручное тестирование, анализ запросов, эмуляция сети |
| **Postman** | Быстрая проверка API вручную |
| **k6** (опционально) | Нагрузочное тестирование |
| **Git** | Контроль версий тестов |

## 8. Автоматизация тестирования
### 8.1 Backend
- Тесты написаны на **pytest**.
- Структура: `unit/` – изолированные тесты эндпоинтов, `integration/` – сквозные сценарии с реальной БД.
- Запуск: `pytest 03-test/unit/ 03-test/integration/ -v --tb=short`.
- Покрытие: успешные ответы, ошибки 4xx/5xx, граничные значения параметров, обязательные заголовки (Idempotency‑Key), поведение при отсутствии авторизации.

### 8.2 Frontend (E2E)
- Реализованы **Selenium WebDriver** тесты на Python для критичных потоков.
- Сценарии: вход → список слотов → бронирование → отмена → выход.
- Хранятся в `e2e/`, запускаются командой `pytest 03-test/e2e/`.

## 9. Критерии начала и завершения тестирования
### 9.1 Критерии начала (Entry Criteria)
- Backend запускается без ошибок.
- Frontend доступен в браузере.
- Тестовые данные присутствуют.
- Все тестовые артефакты (план, чек‑лист, тест‑кейсы) готовы.

### 9.2 Критерии завершения (Exit Criteria)
- Все тест‑кейсы P0 и P1 выполнены, результат — Pass.
- Критические дефекты (P0) отсутствуют.
- Все найденные дефекты зафиксированы в `defect-report.md`.
- Проведено smoke‑тестирование по `checklist.md`.
- Подготовлен итоговый отчёт (`test-report.md`) с метриками и выводами.

## 10. Управление дефектами
- Дефекты фиксируются в `defect-report.md`.
- Формат: ID, описание, серьёзность, шаги воспроизведения, ожидаемый/фактический результат, скриншоты, статус.
- Серьёзность:
  - **Blocker** – полностью блокирует работу.
  - **Critical** – нарушает ключевой сценарий.
  - **Major** – значительное неудобство, есть обходной путь.
  - **Minor** – косметический дефект.
- После исправления дефект проходит верификацию повторным прогоном тест‑кейса.

- Распределение дефектов по серьёзности
- Время выполнения автоматизированного регрессионного набора
- Покрытие требований тест‑кейсами (трассировочная матрица)

## 12. Риски и контрмеры
| Риск | Вероятность | Влияние | Контрмера |
|------|-------------|---------|-----------|
| Недостаток времени на полное E2E‑покрытие | Средняя | Низкое | Приоритет детальным ручным тест‑кейсам, E2E – выборочно по критичным путям |
| Изменение API после написания тестов | Низкая | Высокое | Фиксация контрактов, ежедневная синхронизация с разработчиком |
| Нестабильная работа Selenium | Средняя | Низкое | Предусмотрен fallback на ручное тестирование |
| Отсутствие тестового окружения, идентичного production | Низкая | Среднее | MVP запускается локально, идентичность окружения не критична |

Перед выпуском проводится **приёмочное тестирование** по критериям приёмки (AC‑*), описанным в спецификациях экранов.  
Собирается отчёт о прохождении тест‑кейсов, который подписывается QA‑инженером.

## 16. Роли и обязанности

- **Разработчик** – консультирование, исправление дефектов.
