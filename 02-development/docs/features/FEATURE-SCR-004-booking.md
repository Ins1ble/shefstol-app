# Фича: Оформление записи (SCR‑004)

## 1. Цель
Предоставить клиенту форму для выбора параметров бронирования кулинарного класса:
- Количество мест (себя + до 2 гостей).
- Выбор инвентаря (свой / прокатный) для каждого места.
- Указание пищевых ограничений (опционально).
- Отображение динамически рассчитанной итоговой стоимости.
- Атомарное создание брони с защитой от двойных записей и овербукинга.

## 2. Требования (источники)
- **Функциональные:** FR‑10 (запись на класс), FR‑11 (выбор инвентаря), FR‑12 (до 3 мест), FR‑13 (лимит мест), FR‑14 (учёт проката), FR‑15 (запрет превышения лимитов), FR‑30 (показ цены), FR‑45 (серверный price_total), FR‑50 (пищевые ограничения)
- **Логики:** LOGIC‑002 (расчёт доступности), LOGIC‑003 (расчёт цены)
- **Use Cases:** UC‑1 (запись на класс), включая ошибки E1‑E4
- **User Stories:** US‑5, US‑6, US‑7, US‑8, US‑11, US‑16
- **Экранные требования:** SCR‑004 (дизайн‑бриф `SCR-004-booking.md`)
- **Нефункциональные:** NFR‑2 (≤ 3 экранов до подтверждения), NFR‑8 (без двойной брони), NFR‑9 (идемпотентность повтора)

## 3. Реализация
### 3.1. Файлы
- `frontend/js/booking.js` – функции `startBooking()`, `renderBookingForm()`, `confirmBooking()`, `goBackToSlotCard()`
- `frontend/js/api.js` – вызов `apiPost('/bookings', body, {'Idempotency-Key': key})`
- `frontend/js/utils.js` – форматтеры дат/цены, снек‑бар
- `backend/app.py` – эндпоинт `POST /api/bookings` (атомарная транзакция, проверка лимитов, идемпотентность)

### 3.2. Поток (шаги)
1. **Инициализация формы**
   - Экран открывается из карточки класса (SCR‑003) с параметром `slotId`.
   - Выполняется запрос `GET /api/slots/{slotId}` для получения актуальных данных слота.
   - Инициализируется состояние `currentBooking`:
     - `seats = 1`
     - `equipment[0] = 'rental'` если есть свободные прокатные наборы, иначе `'own'`
     - `dietary = ''`
     - `idempotencyKey = null`

2. **Степпер числа мест**
   - Диапазон: 1 .. `maxSeats`, где `maxSeats = min(free_seats, program.capacity_cap, 3)`.
   - Кнопки «−»/«+» ограничены на границах диапазона.
   - При изменении числа мест:
     - Добавляются/удаляются строки выбора инвентаря.
     - Для новых строк значение по умолчанию: «Прокатный», если лимит проката ещё не исчерпан, иначе «Свой».

3. **Выбор инвентаря**
   - Для каждого места – переключатель «Свой / Прокатный».
   - Опция «Прокатный» становится недоступной, когда `rental_count = free_rental_kits`.
   - Счётчик «Прокатных выбрано: X из Y» обновляется динамически.

4. **Пищевые ограничения**
   - Необязательное текстовое поле (`textarea`).
   - Значение сохраняется в `currentBooking.dietary`.

5. **Динамический расчёт цены**
   - При каждом изменении мест или инвентаря пересчитывается `price_total`:
   - price_total = slot.price × seats_count + slot.rental_price × rental_count
   - - Превью отображается в блоке цены и на кнопке CTA.
- Строка «Прокат» скрывается, если `rental_count == 0`.

6. **Подтверждение бронирования**
- Генерируется UUID `Idempotency-Key` (если ещё не создан для данной попытки).
- CTA переводится в состояние загрузки, форма блокируется.
- Отправляется `POST /api/bookings` с заголовком `Idempotency-Key` и телом `CreateBookingRequest`.
- При успехе (201) – переход на экран успеха BS‑002 с переданным объектом брони.

7. **Обработка ошибок (матрица R‑023)**
| Код ошибки | HTTP | Ситуация | Реакция UI |
|------------|------|----------|------------|
| `slot_full` (есть места) | 409 | Нехватка мест (E1) | Сообщение с числом доступных мест, автоматический откат счётчика до `available_seats`, сброс `idempotencyKey`, перерисовка формы. |
| `slot_full` (нет мест) / `double_booking` | 409 | Гонка или двойная бронь (E3) | Сообщение «Класс заполнен», кнопка «Вернуться к списку» → SCR‑002. |
| нехватка проката | 409 | `rental_count > available_rental_kits` (E2) | Сообщение с числом доступных прокатных наборов, предложение уменьшить прокат или выбрать «Свой». |
| `slot_cancelled` | 410 | Класс отменён студией | Сообщение «Класс отменён и больше недоступен», кнопка «Вернуться к списку». |
| `slot_started` | 422 | Класс уже начался | Сообщение «Класс уже начался — запись недоступна», кнопка «Вернуться к списку». |
| `bad_request` | 400 | Некорректные параметры (защитный сценарий) | Снек «Не удалось оформить запись. Проверьте параметры». |
| `idempotency_key_conflict` | 409 | Тот же ключ с другим телом | Снек «Не удалось оформить запись. Попробуйте ещё раз», сброс ключа. |
| 5xx / сеть | — | Сбой сервера или соединения (E4) | Снек «Не удалось выполнить. Проверьте соединение и повторите», кнопка «Повторить» с тем же `Idempotency-Key`. |

### 3.3. Идемпотентность
- `Idempotency-Key` сохраняется в состоянии `currentBooking`.
- При повторе после E4 используется тот же ключ, чтобы избежать дублирования брони.
- При изменении параметров формы ключ сбрасывается.

## 5. Ручная проверка (чек-лист)
- [ ] Открыть карточку класса → нажать «Записаться». Отображается форма с 1 местом, выбранным инвентарём по умолчанию.
- [ ] Увеличить число мест до 3 – появляются 3 строки выбора инвентаря, цена пересчитана.
- [ ] Выбрать прокатный набор для одного места – в блоке цены появилась строка «Прокат», итог увеличился на `rental_price`.
- [ ] Уменьшить число мест – лишняя строка инвентаря удалена, цена пересчитана.
- [ ] Заполнить поле «Пищевые ограничения» – текст сохраняется в деталях брони после создания.
- [ ] Нажать «Записаться» – бронь создана, открыт экран успеха BS‑002.
- [ ] **Проверка E1**: в бэкенде уменьшить `free_seats` до 1, попробовать записаться на 2 места – появляется сообщение «Недостаточно мест. Свободно: 1», счётчик автоматически уменьшается до 1.
- [ ] **Проверка E3**: авторизоваться другим клиентом, занять последнее место, затем попробовать записаться первым – сообщение «Класс заполнен», кнопка «Вернуться к списку».
- [ ] **Проверка E4**: отключить сеть перед отправкой – появляется снек ошибки и кнопка «Повторить». Включить сеть, нажать «Повторить» – бронь создаётся без дублирования.
- [ ] **Проверка идемпотентности**: сымитировать таймаут, нажать «Повторить» – запрос уходит с тем же ключом, вторая бронь не создаётся.
