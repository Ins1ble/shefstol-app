
# Баг: Зацикливание на экране входа после успешной верификации кода

## 1. Симптом
После того как пользователь вводил правильный одноразовый код, подтверждающий номер телефона, приложение не переходило на главный экран, а **возвращалось обратно к шагу ввода телефона**. Это делало невозможным завершение входа: после каждой попытки ввести код пользователь снова видел поле «Телефон». При этом в консоли браузера и сервера не было явных ошибок, сам запрос `POST /api/auth/verify-code` выполнялся успешно, но последующие защищённые запросы (например, загрузка списка классов) падали с ошибкой 401.

## 2. Требования (источники)
- **LOGIC-001 (OTP-авторизация и сессия)**
  - «После успешного подтверждения кода сервер выдаёт пару JWT-токенов — короткоживущий `access_token` и долгоживущий `refresh_token`, — которые сохраняются в защищённом хранилище…»
  - «При запуске приложение читает пару токенов из защищённого хранилища. Если есть `refresh_token` (и, как правило, валидный `access_token`) — сессия считается активной, экран SCR-001 не показывается…»
- **Критерии приёмки**
  - AC-004: «Дано телефон уже зарегистрирован (`is_new = false`), Когда `verifyAuthCode` вернул 200, Тогда пара токенов (`access_token` + `refresh_token`) сохраняется в защищённое хранилище и сразу открывается SCR-002, шаг 3 пропускается.»
  - AC-005: «Дано номер новый (`is_new = true`) и код подтверждён, Когда пользователь вводит имя (1–100 символов) и нажимает «Продолжить» (`updateProfile` 200), Тогда регистрация завершается и открывается SCR-002.»

## 3. Причина
В функции-обработчике успешной верификации кода (файл `auth.js`, обработчик `verifyBtn.click`) после получения ответа от `POST /api/auth/verify-code` **отсутствовал вызов функции `setTokens()`**, которая сохраняет токены в глобальное состояние и в `localStorage`. Из-за этого:

1. Модуль `api.js` не получал актуальный `access_token` (его свойство `sessionTokens.access` оставалось `undefined`).
2. После перехода на главный экран вызывалась загрузка списка классов (`loadSlots()`), которая отправляла запрос `GET /api/slots` без заголовка `Authorization` (или с `Bearer undefined`).
3. Сервер возвращал `401 Unauthorized`.
4. Интерцептор в `api.js` пытался выполнить тихий рефреш через `POST /api/auth/refresh`, но поскольку валидный `refresh_token` также отсутствовал, рефреш завершался ошибкой.
5. Обработчик ошибки рефреша вызывал `setTokens(null, null)` и принудительно показывал экран входа через `showScreen('login')`, что и создавало эффект «зацикливания».

Таким образом, проблема была вызвана отсутствием синхронизации между успешным ответом сервера и состоянием клиентского модуля API.

## 4. Исправление
В файл `frontend/js/auth.js` в обработчик `verifyBtn.click` после успешного ответа `verifyAuthCode` добавлен вызов:

```javascript
// Сохраняем токены через setTokens, чтобы api.js узнал о них
setTokens(data.tokens.access_token, data.tokens.refresh_token);
```

Этот вызов гарантирует, что:

- `sessionTokens.access` и `sessionTokens.refresh` в `api.js` получают актуальные значения.
- Токены сохраняются в `localStorage` для восстановления сессии при перезагрузке страницы.
- Последующие запросы автоматически включают заголовок `Authorization: Bearer <access_token>`.

После исправления защищённые запросы перестали возвращать 401, и цикличный возврат к экрану входа исчез.
