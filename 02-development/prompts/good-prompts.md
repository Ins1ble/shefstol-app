# Примеры профессиональных промптов для AI-ассистента

Вот несколько выдуманных, но идеально сформулированных промптов, которые могли бы ускорить и улучшить нашу разработку. Они демонстрируют, как правильно ставить задачи AI‑ассистенту.

---

## Промпт 1: Чёткий старт с ограничениями

```markdown
Я начинаю проект веб-приложения для кулинарной студии «Шеф-стол».
Технический стек уже выбран: бэкенд на Python (Flask + SQLite), фронтенд на чистом HTML/CSS/JS (Bootstrap).
Вот полный набор аналитики: бизнес-требования, функциональные требования,
дизайн-брифы экранов и спецификации API.

Создай каркас бэкенда: структуру папок, инициализацию Flask,
модель базы данных с таблицами clients, chefs, programs, slots, bookings.
Напиши seed-данные для двух слотов.
Реализуй первые два эндпоинта: POST /api/auth/request-code и POST /api/auth/verify-code.
Код должен быть в одном файле app.py, но логически разделён комментариями.
```

**Почему это хороший промпт:**
- Конкретный стек и цель.
- Предоставлен контекст (аналитика).
- Перечислены точные артефакты на выходе.
- Ограничение «один файл» упрощает первый запуск.

## Промпт 2: Рефакторинг с конкретными требованиями

```markdown
У нас есть монолитный booking.js, который отвечает за SCR-003, SCR-004, BS-002, SCR-006.
Проведи рефакторинг: выдели каждую зону ответственности в отдельный модуль.
Структура должна быть такой:
- js/screens/slot-card.js (SCR-003)
- js/screens/booking-form.js (SCR-004)
- js/screens/booking-success.js (BS-002)
- js/screens/booking-details.js (SCR-006)
- js/services/booking-api.js (методы createBooking, cancelBooking, getBooking)

Общие функции (formatDate, showSnackbar) оставь в utils.js.
После рефакторинга приложение должно работать без изменений в поведении.
```

**Почему это хороший промпт:**
- Указана текущая проблема (монолит).
- Предложена целевая структура.
- Явное требование сохранения функциональности.

## Промпт 3: Обработка ошибок по матрице

```markdown
В форме бронирования SCR-004 нужно реализовать обработку ошибок от API
согласно матрице R-023. Ошибки должны выводиться не в alert(),
а inline-сообщением под формой или снекбаром.

Вот коды ошибок и требуемое поведение:
- slot_full (available_seats > 0): показать "Недостаточно мест. Свободно: N."
  и автоматически уменьшить счётчик мест до N.
- slot_full (0 мест): показать "Класс заполнен." и кнопку "Вернуться к списку".
- double_booking: показать "У вас уже есть бронь." и кнопку "Мои записи".
- slot_cancelled / slot_started: показать сообщение и кнопку "К списку классов".
- 5xx / сеть: снекбар с возможностью повторить запрос с тем же Idempotency-Key.
```

**Почему это хороший промпт:**
- Полная спецификация поведения.
- Конкретные тексты сообщений.
- Указание, чем заменить alert().
- Учтён Idempotency-Key для повторных попыток.

## Промпт 4: Стабилизация тестов

```markdown
У нас есть 5 интеграционных тестов в test_integration.py.
Два из них падают из-за того, что используют одну in-memory базу данных
и пересекаются по номерам телефонов и ключам идемпотентности.

Перепиши фикстуру app так, чтобы для каждого теста создавалась
отдельная временная база данных (используй tempfile.mkstemp).
Убедись, что каждый тест использует уникальный номер телефона
(генерируй на основе uuid.uuid4().hex).
После исправления все 5 тестов должны проходить стабильно при любом порядке запуска.
```

**Почему это хороший промпт:**
- Описана корневая причина проблемы.
- Предложено конкретное техническое решение.
- Чёткий критерий успеха: все 5 тестов проходят.

## Промпт 5: Документирование дефекта

```markdown
Опиши дефект, который мы нашли и исправили:
зацикливание экрана входа после успешной верификации кода.

Формат: ID, серьёзность, статус, шаги воспроизведения,
ожидаемый/фактический результат, корневая причина, исправление.

Оформи в виде defect-report.md по стандарту IEEE 829.
Добавь выводы: почему этот баг возник и как избежать подобного в будущем.
```

**Почему это хороший промпт:**
- Указан конкретный инцидент.
- Задан формат документа (IEEE 829).
- Требуется анализ первопричины, а не просто описание.

## 📌 Выводы

Эти промпты можно использовать как примеры профессиональной коммуникации с AI‑инструментами. Они чёткие, содержат контекст, ограничения и критерии успеха. Именно так следует ставить задачи, чтобы получать предсказуемый и качественный результат.

### Ключевые принципы хорошего промпта:

1. **Контекст** — укажи стек, проект, имеющиеся артефакты
2. **Цель** — чётко сформулируй, что нужно получить на выходе
3. **Ограничения** — задай рамки (формат, стиль, размер)
4. **Критерии успеха** — как проверить, что задача выполнена
5. **Примеры** — при необходимости покажи ожидаемый формат

### Антипаттерны (чего избегать):

- ❌ «Сделай мне бэкенд» — слишком размыто
- ❌ «Исправь баг» — без описания симптома и контекста
- ❌ «Напиши код» — без указания стека и требований
- ❌ «Улучши это» — без критериев улучшения
