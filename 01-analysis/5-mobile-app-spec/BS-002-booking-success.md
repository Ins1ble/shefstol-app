## Обзор

Экран завершает сценарий записи на кулинарный класс: подтверждает, что бронь успешно создана, даёт клиенту сводку (дата, программа, шеф, места, инвентарь, пищевые ограничения, итоговая цена), напоминает об офлайн-оплате и предлагает дальнейшие шаги — перейти к списку своих записей или вернуться к каталогу классов.

После показа сводки, **только при первой успешной записи**, инициируется системный запрос разрешения на push-уведомления (LOGIC-007). Экран покидается только по явному действию пользователя (кнопки «Мои записи» / «Готово»); автозакрытия по таймеру нет.

Все ошибки записи обрабатываются на SCR-004, поэтому на этом экране состояний Loading/Empty/Error не возникает.

### User Story

> Как клиент, я хочу после подтверждения записи получить однозначное подтверждение успеха и сводку брони, чтобы убедиться, что я записан, и понять, что делать дальше.

### Бизнес-ценность

- Закрытие петли записи позитивным завершением без давления (P6).
- Мягкая подводка к разделу «Мои записи» (SCR-005).
- Единственная точка запроса push-разрешения — в момент очевидной ценности (после первой брони), что повышает конверсию согласия.

## Входные данные

| Название | Тип | Описание |
|----------|-----|----------|
| `booking` | Объект `CreateBookingResponse` | Полные данные созданной брони (содержит `id`, `seats_count`, `rental_count`, `price_total`, `dietary_restrictions`, `slot` с `start_at`, `program.name`, `chef.name`). |

| `reminder_hours` | Массив целых (напр. `[24, 2]`) | Часы напоминаний из ответа `createBooking`. |
| `push_permission_requested` | Локальный флаг | Запрос push уже показывался на устройстве. |

## Инициализация

Экран не выполняет сетевых запросов. Сводка строится из переданного объекта `booking`. Запрос push разрешения — обращение к системному API ОС.

## Элементы экрана

### 1. Блок успеха
Иконка успеха + заголовок «Вы записаны». Состояние передаётся формой и текстом, не только цветом.

### 2. Сводка брони
- Дата/время: `slot.start_at`
- Программа: `slot.program.name`
- Шеф: `slot.chef.name`
- Мест: `seats_count`
- Инвентарь: «свой» / «прокатный» (из `rental_count`)
- Пищевые ограничения: если `dietary_restrictions` не пусто — показывается текст, иначе строка скрыта
- Итого: `price_total` (серверное поле, крупно)
- Напоминание об оплате: «Оплата на месте: наличные или перевод на карту.»

### 3. Кнопки действий
- **«Мои записи»** (primary) → переход на SCR-005
- **«Готово»** (secondary) → возврат на SCR-002

Обе кнопки активны всегда.

### 4. Запрос push-разрешения
После отрисовки сводки, если `is_first_booking = true` и `push_permission_requested = false`:
- Подводка «Напоминания о классе: за 24 ч и 2 ч до начала» (часы из `reminder_hours`)
- Системный диалог запроса разрешения на push
- При отказе — ничего не блокируется, запрос не повторяется

## Критерии приёмки

### Позитивные

| ID | Критерий | P |
|----|----------|----|
| AC-001 | **Дано** успешная запись, **Когда** экран открыт, **Тогда** показана сводка: дата, программа, шеф, места, инвентарь, пищевые ограничения (если есть), итог, офлайн-оплата. | P0 |
| AC-002 | **Дано** первая бронь, **Когда** показана сводка, **Тогда** подводка и системный запрос push. | P1 |
| AC-003 | **Дано** повторная бронь, **Когда** открыт экран, **Тогда** запрос push не показывается. | P1 |

### Негативные

| ID | Критерий | P |
|----|----------|----|
| AC-N01 | **Дано** отказ в push, **Когда** закрыт диалог, **Тогда** кнопки активны, запись не блокируется. | P1 |

### Граничные

| ID | Критерий | P |
|----|----------|----|
| AC-E01 | **Дано** нет пищевых ограничений, **Когда** сводка показана, **Тогда** строка скрыта. | P2 |
| AC-E02 | **Дано** системный «назад», **Когда** нажата, **Тогда** возврат на SCR-002 (эквивалент «Готово»). | P2 |
