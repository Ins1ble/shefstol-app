Цель и контекст

Приложение «Шеф-стол» заменяет ручную запись на кулинарные классы через WhatsApp и Google-таблицу, устраняя двойные брони, путаницу с местами и потерями продуктов. Клиент самостоятельно просматривает расписание, фильтрует, записывается (себя и гостей), выбирает инвентарь (свой или прокатный), отменяет записи, получает напоминания и может указать пищевые ограничения. Платформа — нативное/гибридное мобильное приложение для iOS и Android (mobile-first для использования на кухне и в студии), распространяется через App Store и Google Play (NFR-4). Веб-версии в скоупе нет.

Скоуп — только роль «Клиент». Шеф-повар и владелец работают через существующую инфраструктуру/админку и в приложение не входят. Справочные данные (классы, программы, шефы) — read-only из API; оплата — офлайн (наличные / перевод), приложение лишь показывает цену и фиксирует запись.

Что входит в MVP (принятые решения): - Профиль клиента (FR-47–FR-49): просмотр и редактирование имени и телефона, выход из аккаунта, удаление аккаунта — экран SCR-007. - Цена и прокат (FR-45, FR-11–FR-14): прокат инвентаря (фартук + ножи) — отдельный тариф. Итог записи = цена за место × число мест + тариф проката × число прокатных наборов. Своё оборудование бесплатно. Валюта — рубли (RUB). Итоговую цену считает и возвращает сервер; приложение показывает её, не пересчитывая. - Пищевые ограничения (FR-50): необязательное текстовое поле при бронировании. Шеф получает информацию через существующую систему, клиент видит своё указание в деталях брони. - Адрес студии (FR-9a, NFR-26): отображается текстом в карточке класса. При наличии координат может быть дополнен статической картой; при недоступности карты — текстовый fallback. Тап по адресу открывает шторку BS-004. - Отмена (FR-16–FR-18): только целиком (нельзя отменить часть мест из записи). Правило 2 часов: ранняя отмена освобождает места и прокатные наборы, поздняя — фиксируется без штрафов.

Подробнее о продукте, аудитории, дизайн-принципах и контексте использования — 00-foundations.md §1–2.

Пользовательские сценарии (user flows)

Ключевые пути навигации (полная диаграмма переходов будет детализирована в экранных документах):

  • Вход (трёхшаговый): запуск → SCR-001 (телефон → код из SMS → имя для нового) → SCR-002.
  • Запись (UC-1): SCR-002 → SCR-003 → SCR-004 → BS-002 → (SCR-005). ≤ 3 экранов до подтверждения (NFR-2). После первой записи в BS-002 запрашивается разрешение на push.
  • Фильтрация (UC-3): SCR-002 → BS-001 → SCR-002 (с применёнными фильтрами; множественный выбор типа программы и шефа).
  • Отмена (UC-2): SCR-005 → SCR-006 → BS-003 → SCR-006 (с обновлённым статусом). Отмена — только целиком.
  • Профиль: SCR-007 — просмотр/редактирование имени и телефона, выход, удаление аккаунта; выход/удаление → SCR-001.
  • Адрес студии: SCR-003 / SCR-006 → BS-004 (шторка с адресом и опциональной картой) → возврат на исходный экран.

Тон, бренд, ограничения

  • Тон: простой, прямой, дружелюбный, без давления и штрафов; обращение на «вы» — 00-foundations.md §6.
  • Бренд: не зафиксирован в аналитике — визуальный язык за дизайнером; обязательны лишь структурные ограничения (контраст, размеры, иерархия) — §3.
  • Доступность / mobile-first: крупные тач-зоны, высокий контраст, читаемость на кухне и в студии — §3, §7.
  • Что НЕ делаем на старте (Phase 2+): онлайн-оплата, оценки шефов, автоматическая отмена по форс-мажору, программа лояльности, публичные отзывы, интерфейсы шефа/владельца, SMS-уведомления.

1. Продукт и аудитория

«Шеф-стол» — клиентское мобильное приложение для самостоятельной записи на кулинарные классы студии. Заменяет запись через WhatsApp и Google-таблицу.

Единственная роль — «Клиент». Шеф-повара и владелец в приложение не входят. Справочные данные (классы, программы, шефы) — read-only из API. Оплата — офлайн (наличные / перевод); приложение показывает цену и фиксирует запись, онлайн-оплаты нет.

Контекст использования: клиент находится на кухне или в студии, нередко в спешке, руки могут быть мокрыми. Это диктует крупные элементы, высокий контраст и минимум шагов (NFR-1).

2. Дизайн-принципы

# Принцип Источник Что это значит для макета
P1 Mobile-first для кухни NFR-1 Крупные тач-зоны, высокий контраст, читаемость при ярком освещении, минимум мелкого текста.
P2 Короткий путь к записи NFR-2 От списка до подтверждения — ≤ 3 экранов. Не добавлять необязательных шагов/полей.
P3 Минимальный порог входа NFR-3 Регистрация — только имя + телефон, без пароля. Не запрашивать лишних данных.
P4 Воспринимаемая скорость NFR-6 Скелетоны вместо пустого экрана; отклик списка и подтверждения ощущается < 2–3 с.
P5 Только свои данные NFR-11, NFR-12 Клиент видит лишь свои записи и контакты; в UI нет доступа к чужим/админским данным.
P6 Честность и спокойствие UC-1/UC-2 Ошибки и правила (места, прокат, 2 часа) объясняются понятно и без давления; штрафов нет.

3. Структурные токены (без бренда)

Дизайнер выбирает конкретные значения; ниже — обязательные правила.

3.1 Тач-зоны и размеры

  • Минимальный размер интерактивного элемента — ≥ 44–48 pt по меньшей стороне.
  • Основной CTA — во всю ширину контентной области, высота не менее минимальной тач-зоны.
  • Между кликабельными элементами — отступ, исключающий промахи «мокрым пальцем».

3.2 Контраст и читаемость (NFR-1)

  • Контраст текста к фону — не ниже WCAG AA (обычный текст ≥ 4.5:1, крупный ≥ 3:1).
  • Состояния не передаются только цветом — дублируются иконкой/текстом/формой (важно для яркого освещения и дальтонизма).
  • Важные числа (свободно мест, цена, время старта) — крупные и контрастные.

3.3 Типографическая иерархия (уровни, не шрифты)

  • Заголовок экрана → Заголовок секции/карточки → Основной текст → Вторичный/подпись (caption). Достаточно 4–5 уровней; держать единообразно.
  • Минимальный размер основного текста комфортен для чтения на кухне (не «мелкий серый»).

3.4 Плотность и сетка

  • Единая шкала отступов (например, кратная базовому шагу) — задаёт дизайнер, применяет везде.
  • Контент — в одну колонку (mobile-first), карточки разделены явными отступами/границами.

3.5 Иконки и индикаторы

  • Иконки сопровождаются текстом в ключевых местах (таб-бар, статусы), не несут смысл в одиночку.
  • Индикатор активных фильтров, бейдж статуса записи — визуально считываемы с первого взгляда.

4.2 Таб-бар (авторизованная зона)

Три верхнеуровневых раздела, всегда доступны на корневых экранах: - «Классы» (SCR-002) — стартовая вкладка (список доступных кулинарных классов). - «Мои записи» (SCR-005). - «Профиль» (SCR-007).

Таб-бар скрывается на вложенных экранах (карточка класса, оформление, детали брони) и на bottom sheet. Каждая иконка сопровождается подписью.

4.3 Bottom Sheet (шторки BS-001 / BS-002 / BS-003)

Единые правила для всех шторок: - Высота — по контенту, но не выше ~90% экрана; длинный контент скроллится внутри. - Бэкдроп (затемнение фона) + закрытие по тапу вне шторки (кроме критичных подтверждений, где закрытие — только явной кнопкой). - Swipe-to-close жестом вниз + видимый «грабер» (полоска) сверху. - Явная кнопка закрытия/отмены. Действия-кнопки шторки — в её нижней части. - Открытие/закрытие — плавная анимация снизу вверх.

4.4 Карта навигации

Полная карта переходов будет уточнена в экранных документах. Каждый экран описывает свои входящие/исходящие переходы в разделе «Навигация».

4.5 Адрес студии (текстовый блок, опционально карта)

Сквозной компонент для экранов, где показывается адрес студии (SCR-003, SCR-006). Описывается здесь один раз; экраны на него ссылаются.

┌─────────────────────────────────┐
│ 📍 Адрес студии                  │  ← текстовый блок
│ Лофт на ул. Заводской, 15       │     (адрес из API: studio_address)
│                                  │
│ (опционально — статичная карта)  │  ← если API даёт координаты
│ [ Открыть в картах › ]          │  ← тап → BS-004 или внешнее приложение
└─────────────────────────────────┘
  • По умолчанию: адрес отображается текстом (строка из API, обязательное поле studio_address).
  • Опционально: если API предоставляет координаты, может показываться статичная карта (превью) с пином. Но fallback на текст обязателен.
  • Тап по адресу/карте → открывается шторка BS-004 «Адрес студии» с полным адресом и, при наличии координат, интерактивной картой.
  • Состояния:
  • Loading — скелетон в форме текстового блока (не пустой экран).
  • Error / offline / нет координат — fallback на текст: только адрес строкой; экран не ломается, запись/просмотр остаются доступны.
  • Ключ API карт (если используется) — параметр конфигурации, в макет не зашивается.

6. Tone of voice и общая микрокопия

Тон: простой, прямой, дружелюбный, без жаргона и канцелярита. Обращение на «вы». Сообщения — короткие, по делу, без вины и давления (штрафов в продукте нет).

Сквозные тексты (единые формулировки, переиспользуются экранами):

Контекст Текст
Оплата «Оплата на месте: наличные или перевод на карту.»
Лейблы инвентаря «Свой фартук и ножи» / «Прокатный набор»
Правило отмены «Отмена не позднее чем за 2 часа до начала — место освобождается. Позже — место остаётся за вами, но штрафов нет.»
Поздняя отмена (итог) «Поздняя отмена: место не освобождено (правило 2 часов). Штраф не взимается.»
Кнопка повтора «Обновить»
Сетевая ошибка при загрузке (общая) «Не удалось загрузить. Проверьте соединение и попробуйте снова.»
Сетевая ошибка при действии «Не удалось выполнить. Проверьте соединение и повторите.»
Ошибка сервера при действии (5xx) «Что-то пошло не так. Попробуйте ещё раз позже.»
Ошибка действия без текста от сервера (дефолт 4xx без message) «Не удалось выполнить. Попробуйте ещё раз.»

Числовые лимиты (потолок программы, размер прокатного фонда) не зашиваются в тексты — подставляются из данных класса/программы.

Раздельная модель доступности. Места и прокат считаются независимо: - Места (за раз можно выбрать): max_seats = min(free_seats, program.capacity_cap, 3). - Прокатные наборы: ограничение отдельное — rental_count ≤ free_rental_kits; своё оборудование не расходует прокатный фонд.

Прокатные наборы не ограничивают число доступных мест — формулу доступности «через прокатные наборы» использовать нельзя.

6.1 Каталог снеков успеха (единые формулировки)

Снек успеха показывается после завершённого действия, у которого результат не очевиден из самого перехода. Тон — короткий, утвердительный, без восклицаний. Единые тексты (экраны ссылаются сюда и не вводят свои формулировки):

Действие Экран/Шторка Текст снека успеха Примечание
Сохранение профиля (updateProfile) SCR-007 «Профиль обновлён» —
Подтверждение смены телефона (confirmPhoneChange) SCR-007 «Изменения сохранены» После успешного ввода кода подтверждения.
Удаление аккаунта (deleteAccount) SCR-007 «Аккаунт удалён» Показывается на экране входа после выхода из сессии.
Выход из аккаунта (logout) SCR-007 — (снек не показывается) Обратная связь — сам переход на SCR-001.
Отмена брони (cancelBooking, ранняя) BS-003 → SCR-006 «Бронь отменена» Показывает экран-родитель после закрытия шторки (см. §6.2).
Отмена брони (cancelBooking, поздняя) BS-003 → SCR-006 «Поздняя отмена: место не освобождено (правило 2 часов). Штраф не взимается.» Это успешный исход (см. строку «Поздняя отмена (итог)» выше), а не ошибка.
Создание брони (createBooking) SCR-004 → BS-002 — (снек не показывается) Обратная связь — переход на шторку успеха BS-002 с детальной сводкой. Дублировать снеком нельзя (см. §6.2).
Применение/сброс фильтров (listSlots с фильтрами) BS-001 → SCR-002 — (снек не показывается) Обратная связь — обновлённый список и индикатор активных фильтров.
Успешный pull-to-refresh любой список — (снек не показывается) Обратная связь — обновлённый контент; снек только при ошибке обновления (§6.3).

6.2 Кто показывает снек при закрытии шторки

Когда действие выполняется на шторке (bottom sheet), а её результат нужно показать после закрытия — единое правило: - Снек успеха/итога действия, после которого шторка закрывается, показывает экран-родитель (он остаётся на экране и переживает закрытие шторки). Пример: подтверждение отмены на BS-003 → снек «Бронь отменена» показывает SCR-006. - Снек ошибки действия, при которой шторка остаётся открытой (можно повторить), показывает сама шторка. - Нельзя дублировать обратную связь: если результат уже выражен переходом на отдельную шторку успеха (например, createBooking → BS-002), снек об успехе на экране-инициаторе не показывается.

6.3 Снеки vs Error-заглушка (разграничение)

  • Снеком сообщаются результаты действий (отправка формы, тап CTA, подтверждение) и ошибка при pull-to-refresh (контент на экране сохраняется). Источник текста: 4xx с message → текст из message; 4xx без message → дефолт (§6); 5xx → «Что-то пошло не так…»; сеть → сетевой текст (§6).
  • Error-заглушкой (состояние Error + «Обновить», §5) сообщается провал первичной загрузки данных экрана (5xx/сеть/таймаут), когда показывать нечего.

Единый источник правила отмены. Полный текст правила «2 часов» и формулировки ранней/поздней отмены задаются только здесь (строки «Правило отмены» / «Поздняя отмена»). Экраны SCR-006 и BS-003 ссылаются на эти формулировки и не переписывают их. Граничный случай «ровно 2 часа до старта» трактуется как ранняя отмена (≥ 2 ч → место освобождается).

7. Доступность (NFR-1, NFR-25 — WCAG AA)

Целевой уровень — WCAG 2.1 AA. Обязательные требования: - Контраст: не ниже WCAG AA (обычный текст ≥ 4.5:1, крупный ≥ 3:1) — см. §3.2. - Тач-зоны: интерактивные элементы — ≥ 44 pt по меньшей стороне, с отступами (§3.1). - Dynamic type: поддержка системного увеличения шрифта; layout не «ломается» при крупном тексте (перенос/перекомпоновка вместо обрезки). - Screen reader: все интерактивные элементы имеют текстовую подпись/доступное имя; иконки без подписи получают accessibilityLabel; статусы и важные числа озвучиваются. - Не только цвет: состояния (свободно/нет мест, статус брони, ошибка) дублируются иконкой/текстом/формой, не передаются одним цветом (§3.2) — важно при ярком свете и для дальтонизма. - Малые экраны: макет корректно работает на компактных устройствах (узкая ширина, низкая высота) — контент скроллится, фикс. CTA не перекрывает контент, ничего не обрезается. - Фокус-состояния и обратная связь на тап обязательны. - Адрес студии (§4.5) — не единственный носитель информации: адрес обязательно продублирован текстом. При наличии карты — у неё доступное имя; при недоступности карты текстовый эквивалент остаётся.

8.1 Напоминания и уведомления (FR-33, NFR-13)

  • Заблаговременное напоминание о предстоящем классе; уведомление об отмене класса студией.
  • Канал — push. Доставку обеспечивает существующая инфраструктура; приложение регистрирует push-токен.
  • Запрос разрешения на push показывается после первой успешной записи — на шторке подтверждения BS-002, когда ценность очевидна, а не на старте. Экран входа SCR-001 разрешение не запрашивает.
  • Отдельного экрана/раздела управления уведомлениями в MVP нет (в т.ч. вне профиля SCR-007).

8.2 Безопасность данных в UI (NFR-11, NFR-12)

  • На экранах отображаются только данные текущего клиента.
  • Персональные данные (имя, телефон) не дублируются без необходимости; номер телефона маскируется (например, +7 *** *** ** 67).

8.3 Поведение офлайн и сетевые ошибки (NFR-24, NFR-6)

  • Просмотр кэша офлайн разрешён: ранее загруженные списки/детали показываются из кэша с видимой пометкой устаревания («Данные могут быть неактуальны» / время обновления), а не пустым экраном.
  • Мутации офлайн запрещены: запись, отмена, изменение профиля при отсутствии сети не отправляются и не ставятся в очередь — действие блокируется с понятным сообщением «Нет подключения».
  • Единый паттерн Error / Retry: все сетевые/серверные сбои и таймауты ведут к состоянию Error с кнопкой «Обновить» (для загрузки, §5) или к снеку ошибки с возможностью повтора (для действий, §6.3); тексты — из §6.
  • Таймаут запроса — ~10 с: по истечении показывается ошибка с повтором, экран не «висит».
  • Идемпотентность бронирования: при создании брони клиент передаёт Idempotency-Key; повтор с тем же ключом не создаёт дубль.

1. Назначение и контекст

Экран — единственная точка входа в приложение для неавторизованного пользователя. Его задача (job-to-be-done): дать клиенту попасть внутрь приложения максимально быстро, без обучения и без пароля — подтвердив номер телефона, чтобы тут же перейти к списку классов и записаться на кулинарный класс.

Экран реализует принцип минимального порога входа (см. 00-foundations.md, §2, P3) и обслуживает как первичную регистрацию, так и повторный вход по номеру телефона. Вход — трёхшаговый поток на единой точке входа (без отдельного выбора «регистрация / вход»):

  1. Шаг 1 — Телефон. Клиент вводит номер телефона.
  2. Шаг 2 — Код из SMS (OTP). Клиент вводит код, отправленный на номер; есть таймер повторной отправки.
  3. Шаг 3 — Имя (только для нового пользователя). Если номер ещё не зарегистрирован — клиент указывает имя. Известный номер этот шаг пропускает и сразу попадает в приложение.

2. Пользователь и цель

Кто: клиент кулинарной студии. Часто — новый пользователь без опыта работы с приложением, который открывает его впервые прямо перед классом.

Контекст использования (см. 00-foundations.md, §1): - находится на кухне или в студии, нередко при ярком освещении (нужен высокий контраст); - руки могут быть мокрыми (промахи по мелким элементам); - спешит перед началом класса — нет терпения на длинные формы и обучение.

Цель пользователя: «Хочу записаться на класс — быстро войти и не вспоминать пароль». Цель экрана: провести вход за минимально возможное число действий (номер → код → имя для новых) и сразу отдать пользователя в SCR-002 «Список классов».

6. Компоненты и поведение

Шаг 1 · Поле «Телефон» (phone)

  • Поле с числовой / телефонной клавиатурой и маской ввода национального формата.
  • Маска подсказывает структуру номера и отсекает посторонние символы; пользователь не вводит разделители вручную.
  • Автофокус при открытии экрана.
  • Конкретный формат и допустимые префиксы — параметр конфигурации, не зашивается в макет.
  • По одному и тому же номеру выполняется и регистрация, и повторный вход (уникальное поле).

Шаг 1 · Кнопка «Получить код» (CTA)

  • Во всю ширину контентной области, размер тач-зоны — по 00-foundations.md, §3.1.
  • Disabled: пока телефон не заполнен/не прошёл формат-валидацию (состояние дублируется не только цветом, §3.2).
  • Enabled: телефон валиден по формату.
  • Loading: после нажатия — индикатор, повторные нажатия блокируются; по ответу бэкенда — переход на шаг 2.

Шаг 2 · Поле «Код из SMS» (OTP)

  • Поле с числовой клавиатурой на 4–6 знаков (точная длина — параметр конфигурации, не хардкод). Автофокус при переходе на шаг.
  • Над полем — текст «Мы отправили код на <номер>» с номером из шага 1.
  • Таймер повторной отправки: ссылка «Отправить код повторно» неактивна и показывает обратный отсчёт; по его истечении становится активной и сбрасывает таймер заново при повторе.
  • Кнопка «Подтвердить»: disabled, пока код не введён полностью; loading при проверке; повторные тапы блокируются. По успеху: известный номер → SCR-002; новый номер → шаг 3.
  • Назад в хедере возвращает на шаг 1 (исправить номер) без потери набранного номера.

Шаг 3 · Поле «Имя» (name) — только для нового пользователя

  • Показывается, только если номер не был зарегистрирован ранее. Известный номер этот шаг не видит.
  • Однострочное текстовое поле с лейблом и плейсхолдером-примером, автофокус.
  • Капитализация первой буквы — на усмотрение системной клавиатуры; без принудительных ограничений длины в UI (лимит — из правил валидации, не хардкод).
  • Кнопка «Продолжить»: disabled, пока имя не заполнено/не прошло формат-валидацию; loading при отправке; по успеху — регистрация завершена, открывается SCR-002.

Повторный вход

  • При вводе ранее зарегистрированного телефона после подтверждения кода (шаг 2) пользователь входит в существующий аккаунт без шага 3 (имя уже известно). Отдельного экрана/режима «вход» нет.

Запрос разрешения на push

  • На этом экране не запрашивается. По 00-foundations.md §8.1 разрешение запрашивается после первой успешной записи (в BS-002).

9. Доступность и mobile-first

Базовые правила — 00-foundations.md, §3 и §7. Специфика SCR-001: - Крупные поля и CTA — рассчитаны на ввод «мокрым пальцем» при ярком свете (тач-зоны §3.1, контраст §3.2). - Числовая / телефонная клавиатура для полей «Телефон» (шаг 1) и «Код из SMS» (шаг 2) — без переключения раскладок. - Автофокус на активном поле каждого шага (телефон → код → имя); логичный переход к CTA. - Лейблы и доступные имена у всех полей и кнопок (screen reader, §7 foundations). - Нижний CTA не перекрывается экранной клавиатурой; форма скроллится при необходимости. - Поддержка системного увеличения шрифта без поломки раскладки.

10. Критерии приёмки дизайна

AC-001 — Шаг 1: минимальный ввод телефона - Дано: пользователь не авторизован и открыл приложение - Когда: загрузился экран SCR-001 (шаг 1) - Тогда: видно только поле «Телефон» и кнопка «Получить код»; поля пароля отсутствуют

AC-002 — Шаг 1: CTA активируется при валидном телефоне - Дано: поле «Телефон» пусто - Когда: пользователь ввёл корректный номер - Тогда: кнопка «Получить код» переходит из disabled в enabled

AC-003 — Шаг 2: отправка кода и его подтверждение - Дано: введён валидный телефон - Когда: пользователь нажал «Получить код» - Тогда: открывается шаг 2 с полем кода и таймером повторной отправки; код приходит в SMS

AC-004 — Повторный вход известного номера без шага «Имя» - Дано: телефон уже зарегистрирован ранее - Когда: пользователь подтвердил код на шаге 2 - Тогда: устанавливается сессия и сразу открывается SCR-002 Список классов, шаг 3 пропускается

AC-005 — Новый номер: шаг 3 «Имя» и завершение регистрации - Дано: номер не был зарегистрирован ранее и код подтверждён - Когда: открывается шаг 3, пользователь вводит имя и нажимает «Продолжить» - Тогда: регистрация завершается и открывается SCR-002 Список классов

AC-006 (негативный) — Невалидный телефон блокирует шаг 1 - Дано: пользователь ввёл телефон в неполном/неверном формате - Когда: он пытается получить код - Тогда: CTA остаётся неактивным либо показывается ошибка формата под полем, код не отправляется

AC-007 (негативный) — Неверный код блокирует вход - Дано: пользователь ввёл неверный код на шаге 2 - Когда: он нажимает «Подтвердить» - Тогда: показывается «Неверный код…», вход не выполняется; можно ввести заново или запросить новый код

AC-008 (негативный) — Пустое имя блокирует шаг 3 - Дано: на шаге 3 поле «Имя» пусто - Когда: пользователь нажимает «Продолжить» - Тогда: показывается подсказка заполнить имя, переход не происходит

AC-009 (граничный) — Таймер и повторная отправка кода - Дано: пользователь на шаге 2, код только что отправлен - Когда: идёт обратный отсчёт - Тогда: ссылка «Отправить код повторно» неактивна до конца отсчёта, затем активируется

AC-010 (граничный) — Защита от двойной отправки - Дано: пользователь нажал CTA любого шага - Когда: идёт ожидание ответа и он нажимает кнопку повторно - Тогда: повторная отправка не выполняется; CTA в состоянии Loading

AC-011 (граничный) — Сохранение ввода при ошибке и «Назад» - Дано: на шаге 2 показана ошибка или пользователь нажал «Назад» - Когда: он возвращается/повторяет - Тогда: введённый номер сохранён, повтор без перезаполнения

11. Решения и параметры конфигурации

Открытые вопросы закрыты следующими решениями (см. также 00-foundations.md §8.1): 1. Способ подтверждения входа — SMS-код (OTP). Трёхшаговый поток: телефон → код → имя (для нового). Шаг ввода кода обязателен (§5, §6). 2. Запрос разрешения на push — не на этом экране, а после первой записи (в BS-002). 3. (Параметр конфигурации, не блокер) Формат и допустимые префиксы телефона — задаются конфигурацией; макет использует маску национального формата, конкретные страны/маску уточняет бэкенд. 4. (Параметр конфигурации, не блокер) Длина кода и длительность таймера повторной отправки — параметры конфигурации; в макете показаны как 4–6 знаков и обратный отсчёт.

Требования на дизайн · SCR-002 · Список классов

ID: SCR-002 · Тип: Экран · Зона: АЗ · Приоритет: Critical · Статус: Черновик · Версия: 0.1 · Дата: 2026-06-15

Экранный документ дизайн-требований. Сквозные правила (принципы, токены, каркас, состояния, микрокопия, доступность) описаны в 00-foundations.md и здесь не дублируются — только ссылки и специфика экрана.

Источники: 00-foundations.md · FR-9 · FR-38 · US-2 · US-3 · UC-3

1. Назначение и контекст

SCR-002 «Список классов» — главный экран приложения и стартовая вкладка авторизованной зоны. Показывает каталог доступных кулинарных классов и служит единственной точкой входа в сценарий записи.

По умолчанию отображаются классы на ближайшие 7 дней (дефолт API; больший период — фильтром дат), отсортированные по ближайшей дате старта и сгруппированные по дням. Заполненные классы (свободных мест нет) и отменённые студией слоты не скрываются — они показываются с пометкой «Мест нет» и неактивной кнопкой записи. Фильтр «только со свободными местами» доступен в BS-001, но по умолчанию выключен. Отсюда клиент открывает карточку класса для записи или уточняет выдачу через фильтры.

Экран работает с API в режиме read-only (см. 00-foundations.md §1) и подчиняется сквозному паттерну состояний (00-foundations.md §5).

2. Пользователь и цель

Роль — Клиент. Контекст использования — кухня или студия, нередко яркое освещение, мокрые руки, спешка перед началом класса (см. 00-foundations.md §1).

Цель на экране: быстро просмотреть ближайшие классы и за минимум шагов выбрать подходящий (дата, тип программы, шеф, цена, есть ли места). Список оптимизирован под быстрый выбор: крупные карточки, контрастные ключевые числа, минимум мелкого текста (принципы P1, P2, P4 в 00-foundations.md §2).

3. Навигация

Входящие переходы

  • После успешного входа/регистрации с SCR-001 — это стартовый экран АЗ.
  • По тапу на таб «Классы» в таб-баре (с любого корневого экрана вкладок).
  • Возврат с карточки класса SCR-003 (кнопка «назад») и из сценария записи.
  • Закрытие шторки фильтров BS-001 (Применить / Сбросить / закрытие).

Исходящие переходы

  • Тап по карточке класса → карточка класса SCR-003.
  • Тап по кнопке/иконке «Фильтры» → шторка BS-001.
  • Таб «Мои записи» → SCR-005.
  • Таб «Профиль» → SCR-007.

Таб-бар (три раздела, подписи под иконками, поведение на вложенных экранах) описан в 00-foundations.md §4.2 — ссылаемся, не переописываем.

4. Контент-инвентарь

Каждая карточка класса агрегирует данные класса, программы и шефа:

Элемент Источник данных Назначение в UI
Дата и время начала slot.start_at Когда класс; ключевое число (контрастно).
Название программы (меню) program.name Что за класс.
Тип программы program.type (новичковая / опытная) Подходит ли по уровню; читаемый лейбл, не только цвет.
Шеф chef.name Кто ведёт.
Цена slot.price Стоимость места; ключевое число (контрастно).
Всего мест slot.total_seats Размер группы (M в формулировке «N из M»).
Свободно мест slot.free_seats Доступность; ключевое число (контрастно).

Без хардкода лимитов. Свободные места — данные класса; по сквозному правилу (00-foundations.md §6) доступность мест считается как min(free_seats, program.capacity_cap, 3). Прокатный фонд — отдельная проверка rental_count ≤ free_rental_kits. Конкретные числа подставляются из данных, в макет не зашиваются.

Уровень экрана (не карточки): - Заголовок экрана. - Кнопка/иконка «Фильтры» + индикатор активных фильтров (видим, когда применён хотя бы один фильтр; см. 00-foundations.md §3.5).

6. Компоненты и поведение

6.1 Карточка класса

  • Вся карточка — единая тап-зона (≥ минимальной тач-зоны, 00-foundations.md §3.1), по тапу — переход в SCR-003. Внутри карточки нет конкурирующих кликов.
  • Видимая обратная связь на тап (press-state), фокус для screen reader (00-foundations.md §7).
  • Карточки разделены явными отступами/границами (00-foundations.md §3.4).

6.2 Кнопка/иконка «Фильтры»

  • Размещена в хедере; открывает BS-001.
  • Индикатор активных фильтров — виден, когда применён ≥ 1 фильтр (точка/бейдж/счётчик); считывается с первого взгляда и дублируется не только цветом (00-foundations.md §3.5).
  • Тач-зона ≥ минимальной; подпись/доступное имя обязательны.

6.3 Сортировка и группировка

  • Классы сортируются по ближайшей дате/времени начала (раньше — выше).
  • При длинном списке — заголовки-секции по дням (sticky-разделители «дата»), чтобы клиент ориентировался в расписании. Заголовок дня — не только цвет, но текст.

6.4 Обновление доступности (pull-to-refresh)

  • Жест «потянуть вниз» обновляет список и актуальную доступность мест. Во время обновления — индикатор обновления; контент не очищается без необходимости (воспринимаемая скорость, P4).

6.5 Подгрузка списка

  • В MVP список предстоящих классов грузится целиком (горизонт расписания ограничен — неделя вперёд); специальная пагинация не нужна.
  • (Задел, не блокер MVP.) Если объём вырастет, на API закладывается курсорная пагинация с догрузкой по достижению конца списка и скелетоном в подвале. Включается без изменения макета.

6.6 Класс без свободных мест

  • Класс с free_seats = 0 показывается с явной пометкой «Мест нет» (текст + иконка/форма, не только цвет) и некликабелен — не открывает сценарий записи.
  • Скрыть заполненные классы можно фильтром «только со свободными местами» в BS-001 (по умолчанию выключен).

7. Состояния экрана

Применяется сквозной паттерн (00-foundations.md §5). Специфика экрана:

Состояние Содержимое / Поведение
Loading Скелетоны карточек классов (форма будущих карточек), не пустой экран.
Content Список карточек предстоящих классов (включая заполненные с пометкой «Мест нет»), сортировка по ближайшей дате, секции по дням.
Empty (нет классов) Заглушка: «Пока нет доступных классов» + подсказка зайти позже / обновить.
Empty (нет классов по фильтрам) Заглушка с подсказкой изменить или сбросить фильтры + быстрый доступ к BS-001 (UC-3 A1/E1).
Error Заглушка ошибки + кнопка «Обновить» (общий текст сетевой ошибки — 00-foundations.md §6).

Скорость отклика: загрузка/обновление списка ощущается < 2–3 с (NFR-6, принцип P4); до получения данных показываются скелетоны, а не пустой экран.

8. Валидации и микрокопия

  • Счётчик свободных мест отражает актуальные данные класса — обновляется по pull-to-refresh и при возврате на экран; не кэшируется в устаревшем виде.
  • Формулировка доступности — «Свободно N из M», где N = free_seats, M = total_seats (числа из данных, не зашиты).
  • Класс без мест — пометка «Мест нет» (текст + иконка/форма, не только цвет).
  • Лейбл типа программы — «Новичковая» / «Опытная» (из program.type).
  • Цена и время — из данных класса; формат единый по приложению.
  • Тексты Empty:
  • нет классов вообще — «Пока нет доступных классов. Загляните позже.»;
  • нет по фильтрам — «По выбранным фильтрам ничего не нашлось. Измените или сбросьте фильтры.»
  • Тон — простой и без давления (00-foundations.md §6).

  • Крупные карточки и единая тап-зона на карточку — удобно «мокрым пальцем» на кухне.
  • Ключевые числа контрастны и крупны: свободно мест, цена, время начала (§3.2 foundations).

Источники: 00-foundations.md · FR-38 · UC-3 · US-3

Сквозные правила (тач-зоны, контраст, иерархия, тон, доступность, паттерн состояний, свойства bottom sheet) заданы в 00-foundations.md и здесь не дублируются — документ ссылается на них.

1. Назначение и контекст

Шторка фильтров позволяет клиенту уточнить список классов под свой запрос — не листать весь каталог предстоящих классов, а быстро сузить его до релевантных вариантов (нужная дата, подходящий тип программы, только со свободными местами, конкретный шеф).

Шторка вызывается с экрана SCR-002 «Список классов» и поверх него же применяет результат. Она не выполняет запрос самостоятельно для отображения — итоговая фильтрованная выдача (включая пустой результат) рендерится на SCR-002. BS-001 — это форма выбора параметров, а не отдельный экран результатов.

Контекст использования — кухня/студия, яркое освещение, спешка (см. 00-foundations.md §1): выбор должен быть крупным, быстрым и выполнимым одним пальцем.

2. Пользователь и цель

  • Роль: Клиент (единственная роль приложения).
  • Цель пользователя: «Хочу видеть только подходящие мне классы» — отфильтровать каталог, чтобы быстрее дойти до записи (US-3, NFR-2 «короткий путь к записи»).
  • Задача дизайна: дать собрать комбинацию условий за минимум тапов, понятно показать, что выбрано, и вернуть на список одним действием «Применить».

3. Навигация

Входящие переходы: - Тап по кнопке/иконке «Фильтры» на SCR-002 «Список классов» → шторка открывается снизу вверх поверх списка.

Исходящие переходы: - «Применить» → шторка закрывается, возврат на SCR-002; список перезапрашивается/перефильтровывается с учётом выбранных условий; на SCR-002 отображается индикатор активных фильтров. - «Сбросить» → все параметры возвращаются к значениям по умолчанию (см. §7); при последующем «Применить» SCR-002 показывает дефолтную выдачу (классы на ближайшие 7 дней, only_available=false). - Закрытие (swipe-to-close / тап по бэкдропу / кнопка закрытия) → возврат на SCR-002 без изменения ранее применённых фильтров.

Свойства bottom sheet (бэкдроп, грабер, swipe-to-close, анимация, высота по контенту, скрытие таб-бара) — см. 00-foundations.md §4.3.

4. Контент-инвентарь / Параметры фильтров

Все параметры справочные и read-only приходят из API. Логика комбинирования: «ИЛИ» внутри группы с множественным выбором (тип программы, шеф) и «И» между группами — класс попадает в выдачу, если по каждой заданной группе выполнено хотя бы одно из выбранных значений. Пример: выбраны типы «Новичковая» + «Опытная» и шефы «Анна» + «Игорь» → класс подходит, если (его тип — новичковая ИЛИ опытная) И (его шеф — Анна ИЛИ Игорь).

# Параметр Поле данных Тип значения Поведение
1 Дата / период начала start_at Быстрый пресет или произвольный диапазон (от–до) Пресеты «Сегодня» / «Эта неделя» / «Выходные» + произвольный диапазон; по умолчанию — ближайшие 7 дней (дефолт API; больший период — выбором диапазона).
2 Тип программы program.type Множественный выбор из «новичковая» / «опытная» Чип-лист с мультивыбором: можно выбрать оба типа (ИЛИ внутри группы). По умолчанию — не выбрано (любой).

Числовые лимиты и потолки в фильтрах не зашиваются — критерий «есть места» опирается на данные класса (free_seats), см. 00-foundations.md §6.

6. Компоненты и поведение

6.1 Выбор даты (start_at) — пресеты + диапазон

  • Быстрые пресеты (чипы): «Сегодня», «Эта неделя», «Выходные» — выбирается один; тап заполняет соответствующий диапазон.
  • Произвольный диапазон: два поля «С» и «По»; тап открывает выбор даты (календарь / нативный date picker), удобный одним пальцем. Допускается выбор одной даты (С = По) — фильтр на день.
  • Выбор пресета и ручной диапазон синхронизированы (ручная правка снимает пресет).
  • Прошедшие даты недоступны для выбора (фильтр работает по предстоящим классам).

6.2 Тип программы (program.type) — множественный выбор

  • Набор чипов с мультивыбором: «Новичковая» / «Опытная»; можно выбрать оба (ИЛИ внутри группы).
  • Состояния чипа: невыбран / выбран (дублируется не только цветом — формой/иконкой, 00-foundations.md §3.2).

6.3 Переключатель «Только со свободными местами» (free_seats > 0)

  • Toggle с подписью; по умолчанию выключен (в выдаче и заполненные классы с пометкой «Мест нет»).
  • Состояние считывается без опоры на цвет (положение + текст).

6.4 Выбор шефа (chef_id) — множественный выбор

  • Чип-лист / мульти-селект шефов; можно отметить несколько (ИЛИ внутри группы). Без выбора — «любой».
  • Список — справочные данные из API; на время загрузки — состояние загрузки (см. §7). Загружается отдельным справочным эндпоинтом (GET /chefs или эквивалент).

6.5 Кнопка «Применить» (фиксированная, нижняя, во всю ширину)

  • Состояния: активна (всегда доступна — даже при дефолтных значениях применяет/подтверждает выбор), нажата (feedback), фокус.
  • Тап → закрытие шторки + применение фильтров на SCR-002 (см. §3).

6.6 Кнопка/действие «Сбросить» (в хедере)

  • Состояния: активно (если хоть один параметр отличается от дефолта) / неактивно или скрыто (если всё в дефолте — нечего сбрасывать).
  • Тап → мгновенно возвращает все контролы к значениям по умолчанию внутри шторки (выдача на SCR-002 обновится после «Применить»).

6.7 Свойства шторки

Swipe-to-close, бэкдроп с закрытием по тапу вне, грабер, явная кнопка закрытия, анимация — берутся из 00-foundations.md §4.3, здесь не переописываются.

Требования на дизайн · SCR-003 · Карточка класса

ID: SCR-003 · Тип: Экран · Зона: АЗ · Приоритет: Critical · Статус: Черновик · Версия: 0.1 · Дата: 2026-06-15

Источники: 00-foundations.md · FR-9a · FR-30 · US-4 · UC-1

Сквозные правила (принципы, токены, каркас, паттерн состояний, тон и общая микрокопия, доступность) описаны в 00-foundations.md и здесь не дублируются.

1. Назначение и контекст

Экран показывает все параметры одного кулинарного класса (слота), чтобы клиент мог принять решение о записи и перейти к оформлению. Это промежуточный шаг между списком классов (SCR-002) и оформлением записи (SCR-004) — вложенный экран без таб-бара (см. 00-foundations.md §4.2).

Контекст использования — кухня/студия, яркое освещение, спешка перед началом класса (00-foundations.md §1): клиенту нужно быстро считать ключевые параметры (когда, какая программа, кто ведёт, сколько стоит, есть ли места) и решить, записываться ли.

2. Пользователь и цель

  • Роль: Клиент (единственная роль приложения).
  • Цель пользователя: убедиться, что класс подходит (время, программа, шеф, цена), и понять, есть ли свободные места, чтобы записаться.
  • Задача экрана: дать полную и честную картину класса за один экран и предоставить один очевидный следующий шаг — «Записаться» — либо понятно объяснить, почему запись сейчас невозможна.
  • User story: US-4 — «Как клиент, я хочу видеть все детали класса перед записью, чтобы выбрать подходящий слот».

3. Навигация

Входящие: - Тап по карточке класса в списке SCR-002.

Исходящие: - «Записаться» (основной CTA) → SCR-004 Оформление записи. Активен только при наличии свободных мест (см. §6, §8). - Тап по адресу студии / «Открыть карту» → шторка BS-004 «Адрес студии» (текстовый адрес + опциональная карта, см. 00-foundations.md §4.5). - Назад (в хедере / системный жест) → возврат к списку SCR-002.

8. Валидации и микрокопия

Тон и общие тексты — 00-foundations.md §6.

  • CTA «Записаться» неактивна при free_seats = 0. Пометка рядом/на кнопке: «Мест нет». Никаких штрафов и упрёков (P6).
  • Доступность мест: формулировка вида «Свободно {free_seats} из {total_seats}» — числа подставляются из данных класса, не хардкодятся.
  • Доступность прокатных наборов: показывается числом свободных прокатных наборов (free_rental_kits, из данных, без хардкода); лейблы инвентаря — «Свой фартук и ножи» / «Прокатный набор» (00-foundations.md §6). Точный расчёт лимита применяется на SCR-004.
  • Адрес студии (обязательно): текстовый блок с адресом (studio_address). Например: «Адрес студии: Лофт на ул. Заводской, 15». При наличии координат — статичная карта с пином; тап / «Открыть карту» → BS-004. Текст адреса дублирует карту (доступность, §9).
  • Цена: показывается как цена за одно место (per-seat), значение slot.price из данных. Итоговая сумма за все места считается на SCR-004.
  • Напоминание об офлайн-оплате: текст из 00-foundations.md §6 — «Оплата на месте: наличные или перевод на карту.»

9. Доступность и mobile-first

Полные правила — 00-foundations.md §7, §3.1, §3.2.

  • Ключевые числа (дата/время начала, цена за место, свободно мест) — крупные и контрастные, читаемы на ярком свете.
  • CTA «Записаться» — крупный, во всю ширину контентной области, тач-зона ≥ минимальной.
  • Состояния (тип программы, «Мест нет», «отменено») не передаются только цветом — дублируются текстом/иконкой/формой.
  • Адрес студии не единственный носитель: адрес обязательно присутствует в текстовом виде; при наличии карты у неё есть доступное имя; при недоступности карты текстовый блок остаётся (00-foundations.md §4.5, §7).
  • Все интерактивные элементы имеют доступное имя; layout не ломается при системном увеличении шрифта.

Требования на дизайн · SCR-004 · Оформление записи

ID: SCR-004 · Тип: Экран · Зона: АЗ · Приоритет: Critical · Статус: Черновик · Версия: 0.1 · Дата: 2026-06-15

Экранный документ. Сквозные правила (принципы, токены, каркас, паттерн состояний, tone of voice, доступность) описаны в 00-foundations.md и здесь не дублируются — только ссылки и специфика экрана.

Источники: 00-foundations.md · FR-10–FR-15 · FR-45 · FR-50 · US-5–US-8 · US-16 · UC-1

1. Назначение и контекст

Экран собирает параметры брони (число мест, вариант инвентаря на каждое место, пищевые ограничения, итоговую цену) и фиксирует запись. Это самый сложный экран клиентского сценария: на нём пересекаются два независимых лимита (места группы и прокатный фонд) и обрабатываются конкурентные записи.

Экран — последний шаг перед подтверждением в потоке записи: список → карточка → оформление → подтверждение. Соблюдается правило ≤ 3 экранов до подтверждения (NFR-2): от SCR-002 до BS-002 клиент проходит карточку (SCR-003) и этот экран, без лишних шагов.

Без хардкода. Все числа (free_seats, free_rental_kits, потолок программы, price, rental_price) приходят из данных класса/программы и не зашиваются в макет или тексты (см. 00-foundations.md §6). Правила двух лимитов и формула цены собраны в одном месте — §6.5; остальные разделы ссылаются на него, не повторяя условия.

2. Пользователь и цель

  • Роль: Клиент (единственная роль приложения, см. 00-foundations.md §1).
  • Контекст: клиент на кухне или в студии, спешит, руки могут быть мокрыми — нужны крупные элементы и минимум действий (P1, P2).
  • Цель пользователя: быстро записать себя (и при желании 1–2 гостей), выбрать инвентарь на каждого, указать пищевые ограничения, увидеть итоговую цену и подтвердить бронь.
  • Цель системы: собрать корректную бронь в рамках двух лимитов, не допустить овербукинга и двойной брони, зафиксировать запись.

6. Компоненты и поведение

6.1 Степпер «Число мест» (seats_count)

  • Крупный степпер с кнопками «−» / «+» и числом по центру (тач-зоны ≥ 44–48 pt, 00-foundations.md §3.1).
  • Минимум — 1 (клиент всегда бронирует себя): на минимуме «−» disabled.
  • Максимум — по правилу лимита мест из §6.5 (min(free_seats, потолок программы, 3)). На максимуме «+» disabled (ограничение сверху, без скрытой блокировки) — клиент видит границу.
  • Подпись под степпером поясняет текущий лимит словами (число — из данных, без хардкода).
  • При уменьшении seats_count лишние строки выбора инвентаря (с конца) убираются; при увеличении — добавляются новые строки со значением по умолчанию (см. §6.2).

6.2 Переключатели «Свой фартук и ножи / Прокатный набор» (на каждое место)

  • На каждое место — сегментный переключатель из двух опций: «Свой» / «Прокат» (бинарный, без третьего состояния). Состояние дублируется не только цветом — формой/текстом (00-foundations.md §3.2).
  • Значение по умолчанию для нового места: «Прокат», если rental_count < free_rental_kits; иначе «Свой» (чтобы не создавать заведомо невалидную бронь).
  • Ограничение сверху по прокату: суммарно нельзя выбрать «Прокат» больше, чем free_rental_kits. При достижении границы опция «Прокат» на ещё не-прокатных местах становится недоступной с поясняющей подписью (см. §8, E2), а не молча игнорируется.
  • Подпись-счётчик «Прокатных выбрано: rental_count из free_rental_kits» обновляется в реальном времени.

6.3 Поле «Пищевые ограничения» (dietary_restrictions)

  • Необязательное текстовое поле (FR-50).
  • Плейсхолдер/подсказка: «Укажите аллергии и ограничения заранее».
  • Многострочное (textarea), чтобы клиент мог перечислить несколько пунктов.
  • Сохранённое значение отображается в деталях брони (SCR-006).

6.4 Блок итоговой цены

  • Показывает две строки — места (price × seats_count) и, если есть прокатные места, прокат (rental_price × rental_count) — и итог крупно и контрастно (важное число, 00-foundations.md §3.2). Формула — §6.5.
  • Пересчёт в реальном времени при изменении seats_count и варианта инвентаря (прокат влияет на итог). Строка «Прокат» скрыта при rental_count = 0. Итог дублируется суммой на CTA.
  • Под ценой — напоминание об офлайн-оплате (текст из 00-foundations.md §6: «Оплата на месте: наличные или перевод на карту.»).

6.5 Кнопка «Записаться» (нижний CTA)

  • Во всю ширину контентной области, всегда видна над клавиатурой (00-foundations.md §4.1).
  • Enabled, когда бронь валидна по правилам лимитов из §6.5.
  • Disabled при невалидной комбинации (поясняющая микрокопия рядом, не «глухая» кнопка).
  • Состояние отправки (loading): при тапе CTA переходит в индикатор загрузки и блокируется от повторного тапа (защита от двойной брони / овербукинга, NFR-8 — см. §7).

6.6 Правила лимитов и цены (единый источник, FR-13–FR-15, FR-30)

Лимиты (два независимых):

Лимит Граница Правило
Число мест seats_count 1 ≤ seats_count ≤ min(free_seats, потолок программы, 3) Минимум 1 (сам клиент); максимум — меньшее из свободных мест, потолка программы и 3 (себя + 2 гостя, FR-12).
Прокатные места rental_count rental_count ≤ free_rental_kits Считается из переключателей «Прокат»; не больше свободных прокатных наборов.
  • «Свой фартук и ножи» занимает место в группе, но НЕ уменьшает прокатный фонд (FR-14, UC-1 A1).
  • «Прокатный набор» занимает место в группе И уменьшает прокатный фонд (FR-14, UC-1 A2).
  • Следствие: при free_rental_kits = 0 запись возможна только со своим инвентарём.

Формула итоговой цены:

Итого = price × seats_count + rental_price × rental_count
  • price — цена за место, rental_price — отдельный тариф за один прокатный набор (оба из данных класса).
  • Прокат тарифицируется отдельно, поэтому итог зависит и от числа прокатных мест.

Эти правила — единственное место их описания на экране; §6.1, §6.2, §6.4, §8 ссылаются сюда.

E1 Запрошено мест больше, чем свободно (seats_count > free_seats) «Недостаточно мест. Свободно: [N]. Уменьшите бронь до [N] мест.» Запись отклоняется; показать актуальное free_seats; предложить/выполнить откат счётчика до [N].

10. Критерии приёмки дизайна

AC-001: Отображение формы брони
  Дано клиент перешёл с карточки класса со свободными местами
  Когда открыт экран SCR-004
  Тогда показана сводка класса, степпер мест (1), переключатели инвентаря,
    поле «Пищевые ограничения», блок итоговой цены и кнопка «Записаться»

AC-002: Изменение числа мест и пересчёт цены
  Дано степпер показывает 1 место
  Когда клиент увеличивает число мест до 2 и выбирает прокат для одного из них
  Тогда появляется вторая строка с выбором инвентаря
  И итоговая цена пересчитывается (места × 2 + прокат × 1)

AC-003: Ограничение прокатного фонда
  Дано свободно 2 прокатных набора, выбрано 2 прокатных места
  Когда клиент пытается выбрать «Прокатный набор» для третьего места
  Тогда опция «Прокатный набор» недоступна
  И показана подсказка о лимите

AC-004: Отправка брони и переход к успеху
  Дано форма заполнена корректно
  Когда клиент нажимает «Записаться»
  Тогда CTA переходит в состояние loading и блокируется
  И при успехе открывается BS-002 «Подтверждение записи»

AC-005: Ошибка нехватки мест
  Дано клиент выбрал количество мест, превышающее свободные
  Когда он нажимает «Записаться»
  Тогда показывается ошибка E1 с актуальным числом свободных мест
  И счётчик мест откатывается до доступного максимума

AC-006: Пищевые ограничения сохраняются в брони
  Дано клиент заполнил поле «Пищевые ограничения»
  Когда бронь создана
  Тогда введённый текст сохраняется и отображается в деталях брони (SCR-006)

11. Решения

  • Пищевые ограничения: необязательное текстовое поле при бронировании (FR-50).
  • Формула цены: price_total = price × seats_count + rental_price × rental_count (FR-45).
  • Идемпотентность: клиент передаёт Idempotency-Key при создании брони; повтор с тем же ключом не создаёт дубль.
  • Защита от овербукинга: UI-ограничения предупреждающие, финальная проверка на бэкенде.

2. Пользователь и цель

  • Роль: Клиент.

3. Навигация

  • Кнопка «Перейти к моим записям»: закрывает шторку и переходит на SCR-005.
  • Закрытие шторки без кнопки (свайп, бэкдроп) — возвращает на SCR-002.
  • Запрос push-уведомлений: после первой успешной записи система показывает нативный диалог запроса разрешения на push. Это происходит поверх шторки или сразу после её закрытия — решение за разработчиком, но UX-последовательность: успех → запрос push.

1. Назначение и контекст

Экран даёт клиенту контроль над своими кулинарными классами — отдельно предстоящими и прошедшими. Это домашняя точка для проверки «куда и когда я записан», входа в детали записи и (через детали) запуска отмены. Экран — корневой во вкладке «Мои записи» таб-бара (см. 00-foundations.md §4.2).

Контекст использования — кухня/студия, яркое освещение, спешка перед началом класса (00-foundations.md §1): клиенту нужно с одного взгляда понять ближайшую запись (когда начало, какая программа, сколько мест и какой инвентарь) и её статус.

2. Пользователь и цель

  • Роль: Клиент (единственная роль приложения).
  • Цель пользователя: быстро увидеть свои предстоящие классы, убедиться в деталях ближайшей записи, при необходимости перейти к деталям/отмене; при отсутствии записей — понять, как записаться.
  • Покрываемая история: US-9 — «Как клиент, я хочу видеть список своих записей (предстоящих и прошедших), чтобы контролировать свои классы».

4.2 Разделение списка

  • Предстоящие — только активные записи (status = active), у которых start_at в будущем (ориентир и приоритетная секция).
  • Прошедшие — записи с start_at в прошлом И все отменённые (cancelled / late_cancel / cancelled_by_studio) независимо от даты начала. То есть отменённая запись с будущим началом тоже попадает в «Прошедшие».

Правило отнесения: запись активна и начало в будущем → «Предстоящие»; иначе (начало в прошлом или запись отменена) → «Прошедшие». Активность и время начала вычисляются из данных, без отдельного флага. Прошедшие сортируются по убыванию даты начала (свежие сверху).

# Решение
1 Отменённые записи — всегда в «Прошедшие» (в т.ч. с будущим началом); «Предстоящие» — только активные (§4.2).

Источники: 00-foundations.md · FR-16, FR-17, FR-18, FR-46 · FR-35a · US-10 · UC-2

1. Назначение и контекст

Экран показывает полную информацию об одной записи (брони) клиента и служит точкой запуска её отмены. Это вложенный экран, открываемый из списка «Мои бронирования» (SCR-005) тапом по конкретной записи.

Две задачи экрана:

  • Просмотр. Собрать в одном месте все параметры брони и её текущий статус, чтобы клиент понимал, на что и когда он записан, сколько мест и какой инвентарь, какие пищевые ограничения указаны, какую цену оплатить офлайн.
  • Отмена. Дать единственное действие над записью — отменить участие. Само подтверждение и пояснение последствий (правило 2 часов) выполняется в шторке BS-003, экран лишь запускает её и отображает результат после возврата.

Контекст использования — «кухня/студия, яркое освещение, спешка» (см. 00-foundations.md §1): статус и ключевые числа читаемы с первого взгляда, деструктивное действие защищено подтверждением.

2. Пользователь и цель

  • Роль: Клиент (единственная роль приложения).
  • Цель пользователя: убедиться в деталях своей записи и при необходимости отменить её заранее, понимая последствия.
  • User story: US-10 — как клиент, я хочу видеть детали своей брони и отменять её, чтобы освободить место, если мои планы изменились.
  • Сценарий: UC-2 «Отмена записи» (основной поток, A1 поздняя отмена, исключения E1/E2).
  • Принцип «только свои данные» (00-foundations.md P5): экран показывает данные исключительно текущего клиента.

3. Навигация

Входящие

  • Тап по карточке записи в списке «Мои бронирования» (SCR-005).

  • Кнопка «Отменить» (нижний CTA) → открывает шторку BS-003 «Подтверждение отмены». После подтверждения отмены управление возвращается на SCR-006 с обновлённым статусом.

6. Компоненты и поведение

6.1 Блоки деталей

Сгруппированные секции/карточки (одна колонка, 00-foundations.md §3.4). Каждая секция имеет заголовок-лейбл и значение; ключевые числа (время начала, цена, мест) — крупные и контрастные (00-foundations.md §3.2). Read-only, без редактирования.

6.2 Бейдж статуса

Отражает status записи. Состояния не передаются только цветом — обязательны иконка/текст (00-foundations.md §3.2, §3.5). Возможные значения:

  • Активна — предстоящая, не отменённая запись.
  • Отменена — ранняя отмена (места и прокатные наборы освобождены).
  • Поздняя отмена — отменена < 2 ч до начала (место не освобождено). Текст-итог — из 00-foundations.md §6.
  • Отменён студией — класс отменён организатором (форс-мажор). Дополнительно отображается причина отмены (cancel_reason), если приходит в API.
  • Прошедшая — класс уже начался/завершён.

6.3 Кнопка «Отменить» (нижний CTA)

  • Крупная, во всю ширину (00-foundations.md §3.1).
  • Enabled — только когда запись активна И класс ещё не начался. Тап → открывает BS-003.
  • Disabled (показывается неактивной, не скрывается) — если класс уже стартовал/прошёл (UC-2 E1) либо запись уже отменена (UC-2 E2 — повторная отмена не выполняется). Рядом — обязательное короткое пояснение причины недоступности (см. §8), чтобы причина была читаема.
  • Отмена — только целиком. Частичная отмена отдельных мест в MVP не поддерживается (см. §11).

Деструктивное действие не выполняется на этом экране напрямую — фактическое подтверждение, логика ранней/поздней отмены и текст правила 2 часов — в BS-003.

6.4 Адрес студии

  • Обязательный текстовый блок с адресом (studio_address). При наличии координат — опциональная статичная карта с пином. Тап / «Открыть карту» → BS-004.
  • При недоступности карты — fallback на текст (адрес + ссылка «Открыть в картах»). Состояния — по 00-foundations.md §4.5.

8. Валидации и микрокопия

  • «Отменить» — только до начала (UC-2 E1). CTA доступен, пока slot.start_at в будущем. После начала — недоступен. Пояснение в disabled-состоянии: «Класс уже начался — отменить запись нельзя.»
  • Повторная отмена не выполняется (UC-2 E2). Для записи со статусом «отменена» / «поздняя отмена» / «отменён студией» CTA скрыт/disabled; показывается текущий статус. Пояснение: «Запись уже отменена.»
  • Правило 2 часов — ссылкой, без дублирования. Рядом с активной кнопкой «Отменить» — короткая подсказка на правило отмены; полный текст и логика ранней/поздней отмены — в 00-foundations.md §6 (единый источник) и раскрываются в BS-003. Текст правила здесь не переписывается.
  • Явное время отсечки. Рядом с правилом показываем конкретный момент отсечки — «Бесплатно освободить место можно до <время начала − 2 ч>» (значение вычисляется из slot.start_at, без хардкода). Помогает клиенту сориентироваться, ранняя отмена или поздняя.

Тон — спокойный, без давления и вины (00-foundations.md §6, P6).

9. Доступность и mobile-first

Базовые требования — 00-foundations.md §7 и §3. Специфика экрана:

  • Читаемый статус — бейдж дублирует смысл текстом и иконкой, не только цветом; различим при ярком освещении.
  • Крупный CTA — кнопка «Отменить» во всю ширину, тач-зона ≥ минимальной; disabled-состояние визуально и программно (доступное имя) считывается screen reader.
  • Осторожность с деструктивным действием — отмена не происходит по одному тапу; обязательное подтверждение в BS-003. Кнопка не должна провоцировать случайное нажатие «мокрым пальцем».
  • Layout не ломается при системном увеличении шрифта; ключевые числа остаются крупными и контрастными.

Требования на дизайн · BS-003 · Подтверждение отмены

ID: BS-003 · Тип: Bottom Sheet · Зона: АЗ · Приоритет: High · Статус: Черновик · Версия: 0.1 · Дата: 2026-06-15

Шторка критичного подтверждения деструктивного действия — отмены записи клиентом. Объясняет последствия (ранняя / поздняя отмена по правилу 2 часов) и требует явного подтверждения. Сквозные правила (каркас шторок, состояния, микрокопия, доступность) — в 00-foundations.md; здесь не дублируются, а уточняются под кейс.

Источники: 00-foundations.md · FR-17, FR-18 · UC-2 · US-10

1. Назначение и контекст

Шторка вызывается, когда клиент инициирует отмену предстоящей записи на кулинарный класс и должен осознанно подтвердить деструктивное действие.

Назначение:

  • Объяснить последствия отмены до её выполнения: освобождается ли место в классе и возвращаются ли прокатные наборы.
  • Показать применимость правила 2 часов (см. 00-foundations.md §6) к текущей брони — ранняя или поздняя отмена.
  • Снять тревогу: при поздней отмене штрафов нет (принцип P6 — честность и спокойствие).
  • Получить явное подтверждение и защитить от случайного нажатия (отмена необратима).

Контекст использования — кухня/студия, спешка, мокрые руки (см. 00-foundations.md §1): кнопки крупные, безопасное действие защищено от промаха.

2. Пользователь и цель

  • Роль: Клиент (единственная роль приложения).
  • Цель пользователя: отказаться от участия в классе, понимая последствия, без скрытых санкций.
  • Цель продукта: при ранней отмене вернуть место и прокатные наборы в слот для других клиентов; при поздней — честно зафиксировать, что место и инвентарь за клиентом, и не пугать штрафами.
  • Задача шторки: дать понять разницу между ранней и поздней отменой до подтверждения и предотвратить случайную потерю брони.

3. Навигация

Входящие

  • Кнопка «Отменить» на экране SCR-006 «Детали брони + отмена» (доступна только для активных броней, класс по которым ещё не начался — UC-2 E1).

Исходящие

  • Подтвердить отмену → выполнение отмены (API cancelBooking) → возврат на SCR-006 с обновлённым статусом записи («Отменена» при ранней / «Поздняя отмена» при поздней) и снеком из каталога (00-foundations.md §6.1).
  • Не отменять (отказ) → закрытие шторки, возврат на SCR-006 без изменений.

Свойства шторки

Общие правила bottom sheet — 00-foundations.md §4.3. Уточнение для BS-003 как критичного подтверждения: закрытие — только явной кнопкой («Не отменять» / крестик). Закрытие по тапу вне шторки и по swipe-вниз приравнивается к отказу (безопасное действие), но деструктивное действие невозможно выполнить без явного тапа по кнопке подтверждения.

6. Компоненты и поведение

6.1 Блок текущего случая (две вариации)

Определяется временем до начала (от времени начала класса):

  • РАННЯЯ отмена (≥ 2 ч до начала, FR-17): подтверждение, что место и прокатные наборы возвращаются в слот и снова доступны другим клиентам. Тон нейтральный.
  • ПОЗДНЯЯ отмена (< 2 ч до начала, FR-18): место и прокатные наборы не освобождаются (правило 2 часов), при этом штраф не взимается. Визуально считываемо как пометка/бейдж «Поздняя отмена» (см. 00-foundations.md §5 — специфичный индикатор), смысл не передаётся только цветом (00-foundations.md §3.2).

6.2 Кнопки

  • «Подтвердить отмену» — деструктивное действие. Визуально отлично и менее акцентно относительно безопасной кнопки (стиль destructive — цвет задаёт дизайнер, без хардкода); смысл деструктивности дублируется текстом, не только цветом.
  • «Не отменять» — безопасное действие, основное/акцентное, расположено так, чтобы быть удобным для тапа и не быть «ловушкой» (не маскировать деструктивное под безопасное).
  • Обе кнопки — крупные тач-зоны (00-foundations.md §3.1).

6.3 Состояние выполнения (loading)

После тапа по «Подтвердить отмену» — индикация выполнения: кнопка переходит в состояние загрузки/блокировки, повторные тапы исключены (защита от двойного запроса, NFR-8/NFR-9). Остальные элементы шторки заблокированы до ответа API.

8. Валидации и микрокопия

Точные тексты:

  • Правило 2 часов (из 00-foundations.md §6, переиспользуется без изменений):

  • «Отмена не позднее чем за 2 часа до начала — место освобождается. Позже — место остаётся за вами, но штрафов нет.»

  • Ранняя отмена (≥ 2 ч до начала, FR-17) — подтверждение возврата мест и инвентаря:

  • «Место и прокатные наборы возвращаются в слот и станут доступны другим.»

  • Поздняя отмена (< 2 ч до начала, FR-18, из 00-foundations.md §6):

  • «Поздняя отмена: место и прокатные наборы не освобождены (правило 2 часов). Штраф не взимается.»

  • Ошибка выполнения отмены (общий сетевой текст, 00-foundations.md §6):

  • «Не удалось выполнить отмену. Проверьте соединение и попробуйте снова.»

  • Заголовок: «Отменить запись?»

  • Деструктивная кнопка: «Подтвердить отмену»

  • Безопасная кнопка: «Не отменять»

Время отсечки вычисляется от времени начала класса. Выбор вариации (ранняя / поздняя) — по разнице «время начала − текущее время» относительно порога 2 часа. Числовые лимиты в тексты не зашиваются (00-foundations.md §6).

9. Доступность и mobile-first

Базовые требования — 00-foundations.md §7 и §3.1. Специфика BS-003:

  • Защита от случайного нажатия: деструктивное действие («Подтвердить отмену») требует явного тапа по кнопке; закрытие по бэкдропу/swipe = отказ, а не подтверждение.
  • Крупные кнопки с достаточным отступом между ними («мокрый палец», яркое освещение).
  • Безопасное действие «Не отменять» не должно быть «ловушкой»: не маскируется под деструктивное, имеет понятную акцентную форму и доступное имя для screen reader.
  • Различие ранняя/поздняя передаётся текстом + формой/иконкой, не только цветом (00-foundations.md §3.2).

Требования на дизайн · BS-004 · Адрес студии

ID: BS-004 · Тип: Bottom Sheet · Зона: АЗ · Приоритет: Medium · Статус: Черновик · Версия: 0.1 · Дата: 2026-06-15

Шторка показывает адрес кулинарной студии «Шеф-стол» и, при наличии координат, интерактивную карту с отметкой студии. Вызывается с экранов SCR-003 и SCR-006. Сквозные правила (каркас шторок, состояния, доступность, микрокопия) — в 00-foundations.md; здесь уточняются под этот кейс.

Источники: 00-foundations.md · FR-9a · NFR-26 · US-4

1. Назначение и контекст

Шторка позволяет клиенту увидеть, где находится студия, и при необходимости построить маршрут до неё. Она раскрывает краткий адресный блок с экранов карточки класса (SCR-003) или деталей брони (SCR-006). Если бэкенд передаёт координаты, шторка показывает интерактивную карту с пином студии; если координат нет — только текстовый адрес и кнопки для внешних карт.

Контекст использования — кухня/студия, мобильный телефон, спешка: карта крупная, управление жестами, важная информация (адрес) дублируется текстом и читается при ярком освещении.

BS-004 — вспомогательный просмотр, не критичное подтверждение: закрытие свободным жестом (свайп, тап по бэкдропу) допустимо.

2. Пользователь и цель

  • Роль: Клиент (единственная роль приложения).
  • Цель пользователя: понять, как добраться до студии, увидеть точку на карте, запустить навигацию.
  • Цель продукта: снять неопределённость «куда идти» и предоставить удобный способ построить маршрут (US-4).

3. Навигация

Входящие

  • Тап по текстовому блоку адреса или мини-карте на SCR-003 «Карточка класса».
  • Тап по текстовому блоку адреса или мини-карте на SCR-006 «Детали брони + отмена».

Исходящие

  • Закрытие (кнопка «Закрыть», свайп вниз, тап по бэкдропу) → возврат на исходный экран без изменений.
  • «Проложить маршрут» → открытие внешней навигации (Яндекс.Карты, Apple Maps и т.п.) с построением маршрута до адреса студии.
  • «Открыть в картах» → открытие адреса студии во внешнем картографическом приложении или веб-версии.

Свойства шторки

Общие правила bottom sheet — 00-foundations.md §4.3. Высота шторки — расширенная (до ~90% экрана), чтобы карта была крупной и ей было удобно пользоваться. Поведение закрытия — свободное.

6. Компоненты и поведение

6.1 Текстовый адрес

  • Полный почтовый адрес студии из поля studio_address. Отображается крупно, контрастно, читаем при ярком свете (00-foundations.md §3.2).
  • Служит текстовым эквивалентом для доступности и fallback при отсутствии карты.

6.2 Интерактивная карта (если есть координаты)

  • Отображается, если API передал studio_lat и studio_lng.
  • Полотно карты (Яндекс.Карты или аналог, в зависимости от конфигурации) с поддержкой зума и пана.
  • На карте — пин студии (метка по координатам). По тапу на пин — всплывающая подпись с адресом.
  • Контролы зума (＋/−) — крупные тач-зоны (00-foundations.md §3.1).
  • Ключ API и стиль карты — параметры конфигурации (00-foundations.md §4.5).
  • Если карта не загрузилась (ошибка, нет сети), вместо неё — нейтральное сообщение и кнопка «Обновить». Текстовый адрес и внешние кнопки остаются доступны.

6.3 Кнопки действий

  • «Проложить маршрут» — открывает внешнее навигационное приложение с построением маршрута от текущего местоположения пользователя до координат студии (или до текстового адреса, если координат нет).
  • «Открыть в картах» — открывает координаты/адрес в штатном картографическом приложении устройства (handoff).
  • Обе кнопки — крупные тач-зоны, имеют доступные имена, состояние нажатия.

6.4 Закрытие шторки

  • Кнопка «Закрыть» (в хедере), свайп вниз, тап по бэкдропу — все способы возвращают на исходный экран без изменений. BS-004 — не критичное подтверждение, свободный жест закрытия допустим.

Требования на дизайн · BS-004 · Адрес студии

ID: BS-004 · Тип: Bottom Sheet · Зона: АЗ · Приоритет: Medium · Статус: Черновик · Версия: 0.1 · Дата: 2026-06-15

Шторка показывает адрес кулинарной студии «Шеф-стол» и, при наличии координат, интерактивную карту с отметкой студии. Вызывается с экранов SCR-003 и SCR-006. Сквозные правила (каркас шторок, состояния, доступность, микрокопия) — в 00-foundations.md; здесь уточняются под этот кейс.

Источники: 00-foundations.md · FR-9a · NFR-26 · US-4

1. Назначение и контекст

Шторка позволяет клиенту увидеть, где находится студия, и при необходимости построить маршрут до неё. Она раскрывает краткий адресный блок с экранов карточки класса (SCR-003) или деталей брони (SCR-006). Если бэкенд передаёт координаты, шторка показывает интерактивную карту с пином студии; если координат нет — только текстовый адрес и кнопки для внешних карт.

Контекст использования — кухня/студия, мобильный телефон, спешка: карта крупная, управление жестами, важная информация (адрес) дублируется текстом и читается при ярком освещении.

BS-004 — вспомогательный просмотр, не критичное подтверждение: закрытие свободным жестом (свайп, тап по бэкдропу) допустимо.

2. Пользователь и цель

  • Роль: Клиент (единственная роль приложения).
  • Цель пользователя: понять, как добраться до студии, увидеть точку на карте, запустить навигацию.
  • Цель продукта: снять неопределённость «куда идти» и предоставить удобный способ построить маршрут (US-4).

3. Навигация

Входящие

  • Тап по текстовому блоку адреса или мини-карте на SCR-003 «Карточка класса».
  • Тап по текстовому блоку адреса или мини-карте на SCR-006 «Детали брони + отмена».

Исходящие

  • Закрытие (кнопка «Закрыть», свайп вниз, тап по бэкдропу) → возврат на исходный экран без изменений.
  • «Проложить маршрут» → открытие внешней навигации (Яндекс.Карты, Apple Maps и т.п.) с построением маршрута до адреса студии.
  • «Открыть в картах» → открытие адреса студии во внешнем картографическом приложении или веб-версии.

Свойства шторки

Общие правила bottom sheet — 00-foundations.md §4.3. Высота шторки — расширенная (до ~90% экрана), чтобы карта была крупной и ей было удобно пользоваться. Поведение закрытия — свободное.

6. Компоненты и поведение

6.1 Текстовый адрес

  • Полный почтовый адрес студии из поля studio_address. Отображается крупно, контрастно, читаем при ярком свете (00-foundations.md §3.2).
  • Служит текстовым эквивалентом для доступности и fallback при отсутствии карты.

6.2 Интерактивная карта (если есть координаты)

  • Отображается, если API передал studio_lat и studio_lng.
  • Полотно карты (Яндекс.Карты или аналог, в зависимости от конфигурации) с поддержкой зума и пана.
  • На карте — пин студии (метка по координатам). По тапу на пин — всплывающая подпись с адресом.
  • Контролы зума (＋/−) — крупные тач-зоны (00-foundations.md §3.1).
  • Ключ API и стиль карты — параметры конфигурации (00-foundations.md §4.5).
  • Если карта не загрузилась (ошибка, нет сети), вместо неё — нейтральное сообщение и кнопка «Обновить». Текстовый адрес и внешние кнопки остаются доступны.

6.3 Кнопки действий

  • «Проложить маршрут» — открывает внешнее навигационное приложение с построением маршрута от текущего местоположения пользователя до координат студии (или до текстового адреса, если координат нет).
  • «Открыть в картах» — открывает координаты/адрес в штатном картографическом приложении устройства (handoff).
  • Обе кнопки — крупные тач-зоны, имеют доступные имена, состояние нажатия.

6.4 Закрытие шторки

  • Кнопка «Закрыть» (в хедере), свайп вниз, тап по бэкдропу — все способы возвращают на исходный экран без изменений. BS-004 — не критичное подтверждение, свободный жест закрытия допустим.

7. Состояния экрана

Сквозной паттерн — 00-foundations.md §5 и правила карты §4.5.

Состояние Содержимое / Поведение
Content (с картой) Текстовый адрес, интерактивная карта с пином, кнопки «Проложить маршрут» и «Открыть в картах».
Content (без карты) Текстовый адрес, кнопки действий. Карта скрыта.

8. Валидации и микрокопия

BS-004 не содержит вводимых полей. Контентные тексты:

Кнопка закрытия «Закрыть»
Ошибка загрузки карты «Не удалось загрузить карту. Проверьте соединение и попробуйте снова.» (00-foundations.md §6)
Обновить карту «Обновить»

Тон — простой, без давления (00-foundations.md §6). Адрес и координаты — из данных, без хардкода.

9. Доступность и mobile-first

Базовые правила — 00-foundations.md §7, §3.1, §4.5. Специфика BS-004:

  • Карта не единственный носитель: адрес студии всегда дублируется текстом. При недоступности карты текстовый адрес и внешние кнопки остаются.
  • Крупные контролы (зум, кнопки действий, «Закрыть») с достаточными отступами.
  • Контраст текста и элементов карты — не ниже WCAG AA, читаем при ярком свете.
  • Все интерактивные элементы имеют доступные имена.
  • При увеличении системного шрифта layout не ломается.
