
# «Плохие» промпты (антипримеры)

Примеры промптов, которые привели к неполным, шаблонным или ошибочным результатам. Показаны для контраста.

---

### Антипример 1: Слишком общий запрос

**Промпт:**

```text
Напиши функциональные требования для мобильного приложения кулинарной студии.
```

**Что не так:**
- Нет роли, нет скоупа (только Клиент? шеф? админ?).
- Нет источников (бриф, домен), модель начинает додумывать.
- Нет формата вывода (список? таблица?).
- Результат: получились шаблонные требования вроде «Система должна быть удобной» без конкретики.

**Как исправить:**
- Добавить: «Ты — системный аналитик. Используй бриф customer-brief.md и описание домена domain-description.md. Скоуп — только клиентское мобильное приложение. Формат — таблица с ID, требованием, приоритетом по MoSCoW и источником. Разделы: авторизация, просмотр, запись, отмены, профиль, оплата, уведомления.»

### Антипример 2: Забыты альтернативные потоки

**Промпт:**

```text
Опиши use case «Запись на класс».
```

**Что не так:**
- Не указано, что нужно прорабатывать альтернативные и исключительные потоки.
- Модель выдала только основной поток (7 шагов), полностью проигнорировав ошибки, гонку запросов, нехватку мест.
- Нет привязки к конкретным бизнес-правилам (лимиты, прокат).

**Как исправить:**
- Добавить: «Обязательно проработай альтернативные потоки (A1, A2, A3) и исключения (E1–E5). Укажи бизнес-решения: max_seats = min(free_seats, program.capacity_cap, 3), rental_count ≤ free_rental_kits. Добавь матрицу ошибок с HTTP-кодами.»

### Антипример 3: Игнорирование контекста существующих артефактов

**Промпт:**

```text
Создай дизайн-бриф для приложения записи на кулинарные классы. Опиши экраны.
```

**Что не так:**
- Модель не знает, какие экраны уже согласованы в требованиях (SCR-001...SCR-007, BS-001...BS-004).
- Сгенерировала свои названия экранов («Главная», «Расписание», «Корзина»), которые не совпадают с утверждёнными.
- Не учла наши особенности: поле «Пищевые ограничения» на экране бронирования, адрес студии вместо карты маршрута.

**Как исправить:**
- Добавить: «Используй существующий feature-list.md и функциональные требования FR-9a, FR-50, NFR-26. Скоуп — только роль Клиент. Экраны должны соответствовать SCR-001...SCR-007, BS-001...BS-004. Учти: вместо карты маршрута — адрес студии, на экране SCR-004 — поле «Пищевые ограничения».»
