
# «Хорошие» промпты

Примеры промптов, которые дали качественный, структурный результат с первого раза. Контекст (домен, ограничения, роли) важнее формулировки.

---

### Пример 1: Уточняющие вопросы к брифу

**Промпт:**

```text
Ты — системный аналитик. Прочитай бриф заказчика (файл customer-brief.md) и составь список уточняющих вопросов.
Разбей вопросы на разделы: вместимость, запись, отмены, оплата, аллергии, роли, уведомления, нефункциональные требования.
В каждом разделе — 2–4 вопроса, которые закрывают пробелы и противоречия.
Формат: маркированный список, вопросы от первого лица, как будто задаёшь заказчику.
```

**Почему работает:**
- Чётко задана роль и источник (бриф).
- Указана структура разделов — модель не придумывает что попало, а следует логике предметной области.
- Ограничение на количество вопросов не даёт модели уйти в бесконечную генерацию.
- Формат вывода задан конкретно.

**Фрагмент ответа ИИ:**
- «Вы пишете, что рабочих столов 12, но на классы с духовкой берёте ≤8 человек. По какому признаку класс считается «сложным» — это атрибут программы или вы вручную ограничиваете?»
- «Нужно ли собирать информацию об аллергиях? Если да, на каком этапе — при бронировании или отдельным опросом?»

### Пример 2: Генерация use case'а с альтернативными потоками

**Промпт:**

```text
Создай use case UC-1 «Запись на кулинарный класс» для мобильного приложения.
Актёр: Клиент. Предусловия: авторизован, есть слоты со свободными местами.
Обязательно проработай:

    Основной поток (7 шагов)

    Альтернативные потоки (A1 — все места со своим инвентарём, A2 — часть с прокатом, A3 — фильтр не дал результатов)

    Исключения (E1 — нехватка мест, E2 — нехватка прокатных наборов, E3 — гонка запросов, E4 — сетевой сбой, E5 — слот отменён/начат)

    Матрицу ошибок с HTTP-кодами, деталями и UX-реакцией.
    Используй данные: max_seats = min(free_seats, program.capacity_cap, 3), rental_count ≤ free_rental_kits.
```

**Почему работает:**
- Задана конкретная структура (основной поток, альтернативные, исключения, матрица) — модель не забывает альтернативные пути.
- Указаны бизнес-правила (лимиты, раздельный учёт), которые должны быть отражены.
- Ограничение по количеству шагов и исключений не даёт модели уйти в абстракцию.

**Фрагмент ответа ИИ:**
- Основной поток: «1. Клиент открывает список классов → 2. Применяет фильтры → 3. Открывает карточку → 4. Указывает число мест и инвентарь → 5. Подтверждает запись → 6. Система проверяет max_seats и rental_count → 7. Показывает подтверждение с ценой.»
- Матрица ошибок: `slot_full` → 409, `double_booking` → 409, `slot_cancelled` → 410, `slot_started` → 422, с колонками «Обязательные details» и «UX-реакция».

### Пример 3: Спецификация экрана по шаблону

**Промпт:**

```text
Создай спецификацию экрана SCR-004 «Оформление записи» для приложения «Шеф-стол» по шаблону _SCREEN_TEMPLATE.md.
Используй существующие артефакты: FR-10–FR-15, FR-45, FR-50, UC-1, US-5–US-8, US-16, LOGIC-002, LOGIC-003.
Обязательные разделы: Обзор, Навигация, Входные данные, Применяемые логики, Инициализация, Используемые запросы (createBooking, getSlot), Макет экрана, Элементы экрана, Состояния экрана, Действия пользователя, Критерии приёмки.
В запросе createBooking обязательно описать заголовок Idempotency-Key и все возможные ответы (201, 409, 410, 422, 5xx) с реакцией UI.
```

**Почему работает:**
- Жёсткая привязка к существующим артефактам (FR, UC, US, LOGIC) гарантирует согласованность.
- Перечень обязательных разделов не даёт модели что-то пропустить.
- Указание конкретных деталей (Idempotency-Key, коды ответов) заставляет модель проработать краевые случаи.
- Ссылка на шаблон _SCREEN_TEMPLATE.md обеспечивает единую структуру.
