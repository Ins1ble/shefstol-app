# Функциональные требования

Документ описывает поведение и функции клиентского мобильного приложения кулинарной студии «Шеф-стол». Включает требования к авторизации, просмотру и фильтрации расписания, бронированию, управлению записями, профилю, оплате и уведомлениям. Не путать с нефункциональными требованиями (non-functional-requirements.md).

Приоритеты по MoSCoW: **Must** — обязательно к запуску; **Should** — желательно при наличии времени; **Could** — приятное дополнение; **Won't** — вне границ скоупа.

**Нумерация требований.** ID требований (FR-NN) стабильны для трассировки. Пропуски в нумерации (например, FR-3–FR-8, FR-19–FR-29 и др.) — следствие сознательного сужения скоупа: часть ранее заведённых требований вынесена в развитие/Phase 2 или относится к существующей инфраструктуре. Нумерация намеренно не нормализуется для сохранения связей со смежными артефактами. Новые требования продолжают сквозную схему FR-NN без переиспользования занятых номеров.

**Границы скоупа (клиентское приложение + API).** Скоуп включает только клиентское мобильное приложение и API для него. Функции шефа, владельца и администратора обеспечиваются существующей инфраструктурой студии и здесь не описываются. Слоты (классы), программы (меню) и шефы поступают из существующего бэкенда через API и являются для клиента read-only данными — приложение их потребляет, но не создаёт и не редактирует.

## 2. Просмотр и фильтрация слотов (клиент)

| ID | Требование | Приоритет | Источник |
|----|------------|-----------|----------|
| **FR-9** | Система должна показывать клиенту список предстоящих классов на ближайшие 7 дней от текущего момента (дефолт API: date_from = now, date_to = now + 7 дней; больший период — явным фильтром дат). Состав карточки в списке: дата/время начала, программа (меню) и её тип, шеф, всего мест, свободно мест, цена. Заполненные (свободно = 0) и отменённые студией слоты показываются с пометкой «Мест нет» и неактивной кнопкой записи (CTA). По умолчанию список не отфильтрован (only_available=false). Сортировка — по времени начала (ближайшие выше). При отсутствии слотов — empty state «Пока нет доступных классов». | Must | customer-questions №6–7; domain-description; R-027 |
| **FR-38** | Система должна позволять клиенту фильтровать список слотов по: дате/периоду старта, типу программы (новичковая/опытная), шефу, а также опционально показывать только слоты со свободными местами («только свободные» — необязательная опция, по умолчанию выключена). Фильтр по дате включает быстрые пресеты «Сегодня / Эта неделя / Выходные» и кастомный диапазон («с / по»). Пресеты и диапазон клиент маппит в параметры запроса date_from/date_to. | Must | customer-questions №7; R-012 |
| **FR-9a** | Система должна показывать клиенту карточку слота со всеми параметрами: дата/время, программа (меню) и её тип, описание программы, шеф, цена, всего/свободно мест, доступность прокатных наборов (свободный прокатный фонд), адрес студии. | Must | customer-questions №6; domain-description |

**Семантика фильтров (FR-38).** Внутри одной группы значения объединяются по ИЛИ (OR): program_type и chef_id — массивы значений (например, выбраны и новичковая, и опытная программа → показать обе). Между разными группами фильтров действует И (AND) (например, тип программы И период дат И шеф). Границы периода дат — включительные; даты вводятся в локальной дате клиента. По умолчанию фильтры не заданы (показывается дефолтная выдача — слоты на ближайшие 7 дней, см. FR-9). Если справочник шефов пуст, фильтр по шефу скрывается. Активные (применённые) фильтры отображаются клиенту.

## 3. Запись на класс (клиент)

| ID | Требование | Приоритет | Источник |
|----|------------|-----------|----------|
| **FR-10** | Система должна позволять клиенту записаться на выбранный слот. | Must | Бриф; domain-description |
| **FR-11** | Система должна позволять клиенту при записи выбрать для каждого участника вариант инвентаря: свой или прокатный. | Must | Бриф; customer-questions №3–4 |
| **FR-12** | Система должна позволять клиенту одной записью забронировать несколько мест — не более 3 мест на одну бронь (себя + до 2 гостей) — при наличии свободных мест. По гостям приложение собирает только количество мест и выбор инвентаря (свой/прокат) на каждое место. Имена, контакты и возраст гостей в MVP не собираются; ответственность за гостей несёт клиент, оформивший бронь. | Must | customer-questions №5; R-013 |
| **FR-13** | Система должна ограничивать число доступных к бронированию мест слота величиной max_seats = min(free_seats, program.capacity_cap, 3), где free_seats — свободные места слота, program.capacity_cap — потолок программы (новичковая ≤8, опытная ≤12), 3 — лимит мест на одну бронь. | Must | customer-questions №1; domain-description |
| **FR-14** | Система должна вести отдельный учёт прокатного фонда независимо от учёта мест: число прокатных наборов в брони ограничено свободным прокатным фондом — rental_count ≤ free_rental_kits. Запись «своё оборудование» занимает место в группе, но не занимает прокатный набор из фонда. | Must | customer-questions №3; domain-description |
| **FR-15** | Система должна запрещать запись, превышающую лимит мест (FR-13) или лимит свободных прокатных наборов (FR-14), исключая двойную бронь и овербукинг. | Must | Бриф (двойные записи); customer-questions №3 |
| **FR-45** | Система должна использовать price_total из ответа API как итоговую стоимость брони (read-only; сервер — источник истины). Приложение не пересчитывает итог самостоятельно: при расхождении локальной оценки и серверного значения приоритет имеет price_total из ответа. | Must | customer-questions №15; R-010 |
| **FR-50** | Система должна позволять клиенту при бронировании указать пищевые ограничения — необязательное текстовое поле «Пищевые ограничения» (dietary_restrictions). | Should | customer-questions №21–23 |

**Граница со серверной интеграцией.** Детали серверной реализации брони (транзакции, SLA, маппинг полей) — вне скоупа клиентского приложения. Существующий бэкенд выступает black-box источником истины; атомарность операций и гарантия «0 двойных броней» обеспечиваются на его стороне. Граница между приложением и бэкендом закрывается контрактом API; клиент полагается на коды ответов и значения полей из ответа (включая max_seats, free_rental_kits, price_total).

**Модель цены (учёт).** Итоговая стоимость брони рассчитывается сервером по модели:  
`price_total = price × seats_count + rental_price × rental_count`  
где `price` — цена одного места на слоте, `seats_count` — число мест в брони, `rental_price` — цена проката одного набора (фартук + ножи), `rental_count` — число прокатных наборов в брони. Своё оборудование бесплатно; прокат оплачивается отдельным тарифом. Валюта — RUB. Цена фиксируется на момент создания брони и далее не меняется. Приложение отображает итог через `price_total` из ответа API (см. FR-45), не пересчитывая его локально.

## 4. Мои бронирования и отмены (клиент)

| ID | Требование | Приоритет | Источник |
|----|------------|-----------|----------|
| **FR-35a** | Система должна показывать клиенту всю историю его бронирований с пагинацией (limit/offset), со статусом, параметрами слота, числом мест и вариантом инвентаря. История сортируется по slot.start_at; разбиение на «предстоящие» и «прошедшие» выполняется на клиенте. В списке отображаются в том числе отменённые брони и поздние отмены (late_cancel). При удалении аккаунта история анонимизируется (см. FR-48). | Must | customer-questions №27; domain-description |
| **FR-16** | Система должна позволять клиенту отменить свою запись до начала класса. Отмена возможна только целиком: при отмене освобождаются все места и все прокатные наборы этой брони. Частичная отмена отдельного места или гостя не поддерживается. | Must | customer-questions №10; domain-description |
| **FR-17** | Система должна при ранней отмене (не позднее чем за 2 часа до начала) освобождать забронированные места и прокатные наборы обратно в слот. Граница «ранней» отмены — ровно за 2 часа до начала включительно считается ещё ранней (бесплатной) отменой. Решение о типе отмены принимает сервер как источник истины по времени и статусу. | Must | customer-questions №9; domain-description |
| **FR-18** | Система должна при поздней отмене (менее чем за 2 часа до начала) фиксировать запись статусом «поздняя отмена» (late_cancel) без освобождения места и без денежных штрафов. | Must | customer-questions №10; domain-description |
| **FR-46** | Система должна отображать брони, отменённые студией (форс-мажор: срыв поставки, поломка оборудования): бронь не исчезает из истории, а переходит в статус «Отменён студией» с указанием причины. Клиенту приходит push-уведомление об отмене. Повторная запись на отменённый студией слот запрещена — попытка возвращает HTTP 410. В списке и деталях брони отображаются статус «Отменён студией» и причина; кнопка записи на такой слот неактивна. | Must | customer-questions №31; R-008 |

**Работа со временем.** Время хранится и передаётся в API в UTC, а отображается клиенту в его локальной часовой зоне. Сервер — источник истины по времени и по статусу отмены (ранняя / поздняя / отменена студией). Граница ранней отмены — ровно за 2 часа до начала включительно (см. FR-17).

## 7. Уведомления

| ID | Требование | Приоритет | Источник |
|----|------------|-----------|----------|
| **FR-33** | Система должна напоминать клиенту о предстоящей записи через системные push-уведомления (канал MVP — push). ## Вне скоупа

Функции шефа, владельца/администратора, расписание-CRUD, онлайн-оплата, автоматический учёт форс-мажоров и программа лояльности обеспечиваются существующей инфраструктурой или относятся к развитию — в клиентском приложении и его API не реализуются.

**Оценки шефов — Phase 2** (осознанное сужение скоупа, подтверждено заказчиком). Оценки шефов не входят в MVP и переносятся в Phase 2. Это сознательное решение: заказчик ранее называл функцию критичной, поэтому она не отбрасывается, а планируется к реализации после запуска критичного набора MVP (просмотр слотов, запись, отмены) в жёсткий срок начала сезона. Перенос согласован с заказчиком. На MVP-скоуп и трассировку требований это не влияет.

**«Поделиться» классом — Won't (MVP) / Phase 2** (согласовано с заказчиком). Функция «поделиться» слотом (share класса) не входит в MVP и переносится в Phase 2. На карточке класса иконка «Поделиться» в MVP неактивна/декоративна; специфицировать поведение шеринга в MVP не требуется.
