# Use cases / сценарии

Документ описывает ключевые сценарии использования клиентского мобильного приложения кулинарной студии «Шеф-стол». Все сценарии ограничены ролью «Клиент». Расписание (слоты), программы (меню) и шеф-повара поступают из существующей инфраструктуры через API и доступны только для чтения. Создание, редактирование и отмена слотов целиком выполняются в административной части, вне зоны ответственности клиентского приложения.

---

## UC-1: Запись на кулинарный класс (себя и гостей)

**Актёры:** Клиент  
**Предусловия:** Клиент авторизован; в расписании на ближайшие 7 дней есть хотя бы один класс со свободными местами.  
**Триггер:** Клиент хочет забронировать одно или несколько мест на конкретный класс.

**Бизнес-решения:**
- **Состав гостей (R-013):** одна бронь может включать до трёх мест (сам клиент + до 2 гостей). По гостям собирается только количество мест и выбор инвентаря (свой / прокатный). Имена, контакты и другие персональные данные гостей не запрашиваются. Ответственность за гостей несёт клиент, оформивший бронь.
- **Идемпотентность (R-022):** при создании брони клиент обязан передавать уникальный ключ идемпотентности (Idempotency-Key). Повторный запрос с тем же ключом и телом возвращает идентичный ответ и не приводит к дублированию брони. Запрос с тем же ключом, но другим телом, считается ошибкой.
- **Время и источник истины (R-021):** всё время хранится и передаётся в UTC, отображается в локальной часовой зоне клиента. Сервер является единственным источником истины по времени и статусам; при расхождении клиентской оценки и ответа сервера применяется серверный статус.
- **Пищевые ограничения:** клиент может опционально указать текстовую информацию об аллергиях и диетических предпочтениях (поле `dietary_restrictions`). Эти данные передаются шефу через бэкенд.

### Основной поток

1. Клиент открывает список классов (по умолчанию — на ближайшие 7 дней, все слоты, включая заполненные, которые отображаются с пометкой «Мест нет» и неактивной кнопкой записи).
2. (Опционально) Применяет фильтры: дата/период, тип программы (новичковая/опытная), конкретный шеф, «только свободные».
3. Открывает карточку интересующего класса, видит: дату/время, программу (меню) и её описание, шефа, цену за место, общее и свободное количество мест, доступный прокатный фонд, адрес студии.
4. Указывает количество мест (от 1 до 3). Для каждого места выбирает вариант инвентаря: «свой» или «прокатный». При необходимости вводит пищевые ограничения в отдельное необязательное поле.
5. Нажимает кнопку подтверждения бронирования. Приложение формирует и отправляет запрос с уникальным Idempotency-Key.
6. Сервер проверяет:
   - лимит мест: `max_seats = min(free_seats, program.capacity_cap, 3)`, где `free_seats` — актуальное число свободных мест, `program.capacity_cap` — потолок программы (8 для новичковой, 12 для опытной);
   - доступность прокатных наборов: `rental_count ≤ free_rental_kits`;
   - слот всё ещё активен (не отменён студией, не начался).
7. При успехе создаётся бронь, счётчики свободных мест и прокатного фонда уменьшаются, а запись добавляется в «Мои бронирования».
8. Клиенту показывается экран подтверждения с итоговой стоимостью (`price_total`), составом брони и напоминанием об офлайн-оплате.

### Альтернативные потоки

- **A1. Все места со своим инвентарём:** на шаге 4 для всех мест выбран вариант «свой». Прокатный фонд не затрагивается, уменьшается только количество свободных мест в слоте.
- **A2. Часть мест с прокатом:** на шаге 4 часть мест отмечена как «прокатные». На шаге 6 сервер проверяет и списывает только указанное количество прокатных наборов; места, занятые «своим» инвентарём, на прокатный фонд не влияют.
- **A3. Фильтр не дал результатов:** на шаге 2 после применения фильтров список классов пуст. Система показывает сообщение «Нет классов, соответствующих фильтрам» и предлагает изменить или сбросить фильтры.

### Исключения / ошибки

- **E1. Нехватка мест:** запрошено больше мест, чем доступно (`slot_full`). Бронь отклоняется, клиент видит актуальное количество свободных мест и может скорректировать запрос.
- **E2. Нехватка прокатных наборов:** запрошено больше прокатных комплектов, чем есть в наличии. Система сообщает об этом и предлагает уменьшить число прокатных мест или переключить часть на «своё оборудование».
- **E3. Гонка запросов / слот заполнился параллельно:** другой клиент занял последние места в момент подтверждения. Бронь отклоняется (`slot_full`), интерфейс обновляет информацию о доступности.
- **E4. Сетевой сбой / неопределённый результат:** при обрыве соединения или таймауте клиент повторяет запрос с тем же Idempotency-Key. Повторная отправка не создаёт дублирующую бронь (NFR-9). Мутация офлайн недоступна: при отсутствии сети кнопка подтверждения заблокирована с соответствующим сообщением.
- **E5. Слот отменён или уже начался:** если к моменту подтверждения класс был отменён студией (`slot_cancelled`, HTTP 410) или уже стартовал (`slot_started`, HTTP 422), система отклоняет запись с соответствующим кодом и описанием.

### Матрица ошибок ключевых операций (R-023)

Таксономия ошибок основана на кодах API. Локализованные сообщения формирует клиентское приложение, опираясь на код и поля `details`.

| Код ошибки | HTTP-статус | Обязательные details | UX-реакция | Возможность повтора (Retry) |
|------------|-------------|----------------------|------------|------------------------------|
| `slot_full` | 409 | `available_seats`, `available_rental_kits` | Показать актуальное число свободных мест и прокатных наборов; кнопка повтора недоступна, требуется изменить количество/тип мест | Нет (изменить запрос) |
| `double_booking` | 409 | `booking_id` | Сообщить, что бронь на этот слот уже существует; предложить перейти в «Мои бронирования» | Нет |
| `slot_cancelled` | 410 | `slot_id` | Сообщить, что класс отменён студией; кнопка записи деактивируется | Нет |
| `slot_started` | 422 | `slot_id`, `start_at` | Сообщить, что класс уже начался или завершён; запись невозможна | Нет |
| `invalid_code` | 400/401 | — | «Неверный код подтверждения»; поле ввода OTP активно | Да (ввести код заново) |
| `unauthorized` | 401 | — | Сессия истекла или недействительна; переход на экран авторизации | Да (после повторного входа) |
| `forbidden` | 403 | — | Сообщение об отсутствии прав; операция недоступна | Нет |
| `rate_limit` | 429 | `retry_after` | «Слишком много попыток. Повторите через X секунд»; кнопка заблокирована на указанное время | Да (после задержки) |
| `server_error` | 5xx | — | Единое сообщение «Что-то пошло не так. Попробуйте позже»; для брони — возможен повтор с тем же Idempotency-Key | Да |

### Постусловия

- **При успехе:** создана новая бронь; счётчики свободных мест и прокатного фонда обновлены; запись доступна в разделе «Мои бронирования».
- **При ошибке:** состояние слота не изменилось.

## UC-2: Отмена записи

**Актёры:** Клиент  
**Предусловия:** У клиента есть хотя бы одна предстоящая бронь (класс ещё не начался).  
**Триггер:** Клиент решает отказаться от участия в классе.

**Бизнес-решения:**
- **Отмена только целиком (R-014):** отменяется вся бронь со всеми местами и прокатными наборами. Частичная отмена (например, только одного гостя) невозможна.
- **Время и источник истины (R-021):** сервер определяет, является ли отмена ранней или поздней, исходя из оставшегося времени до начала класса. Граница политики: ровно за 2 часа до начала — ещё ранняя (бесплатная) отмена. При расхождении клиентского прогноза и ответа сервера применяется серверный статус.

### Основной поток

1. Клиент открывает раздел «Мои бронирования» и выбирает нужную предстоящую запись.
2. Нажимает кнопку «Отменить». Приложение отображает предупреждение с информацией о правилах отмены (целиком, политика 2 часов) и запрашивает подтверждение.
3. Клиент подтверждает действие.
4. Сервер вычисляет время до начала класса и тип отмены:
   - **Ранняя отмена (≥2 ч):** бронь помечается как «отменена», все занятые места и прокатные наборы возвращаются в слот.
   - **Поздняя отмена (<2 ч):** бронь получает статус «поздняя отмена» (`late_cancel`), места и прокатные наборы **не** освобождаются. Штрафы не начисляются.
5. Приложение отображает результат: для ранней отмены — «Бронь отменена», для поздней — пояснение, что место не освобождено, но санкций нет.

### Альтернативные потоки

- **A1. Поздняя отмена (<2 ч):** на шаге 4 сервер возвращает статус `late_cancel`. Клиенту показывается информация о том, что место остаётся за ним, продукты уже закуплены, но дополнительная плата не взимается.
- **A2. Расхождение локального времени и серверного:** клиентский интерфейс мог предполагать раннюю отмену, но сервер зафиксировал позднюю (или наоборот). В любом случае действует серверный статус; клиент видит актуальное состояние брони.

### Исключения / ошибки

- **E1. Класс уже начался или завершён:** кнопка отмены недоступна, вместо неё отображается статус «Класс уже прошёл». Попытка отправить запрос через API возвращает ошибку `slot_started` (422).
- **E2. Бронь уже отменена:** запрос на повторную отмену возвращает ошибку `already_cancelled`. Интерфейс отображает текущий статус без изменений.
- **E3. Сетевой сбой / неопределённый результат:** при отсутствии сети кнопка отмены заблокирована с сообщением «Нет подключения». При возникновении ошибки в процессе применяется единый паттерн Error/Retry; финальный статус брони всегда получается с сервера. Коды ошибок и поведение соответствуют общей матрице (UC-1).

### Постусловия

- Статус брони обновлён («отменена» или «поздняя отмена»).
- При ранней отмене все занятые места и прокатные наборы снова доступны другим клиентам.

## UC-3: Просмотр и фильтрация списка классов

**Актёры:** Клиент  
**Предусловия:** Клиент авторизован.  
**Триггер:** Клиент хочет найти подходящий кулинарный класс.

### Основной поток

1. Клиент открывает вкладку «Классы» (основной список классов). По умолчанию отображаются **все** слоты на ближайшие 7 дней (параметры API: `date_from = now, date_to = now + 7 days, only_available = false`), отсортированные по времени начала (ближайшие сверху).
2. Заполненные (0 свободных мест) и отменённые студией слоты остаются в списке, но помечаются индикатором «Мест нет». Кнопка записи для них недоступна.
3. (Опционально) Клиент открывает панель фильтров и задаёт один или несколько критериев:
   - дата/период (пресеты «Сегодня», «Эта неделя», «Выходные» или произвольный диапазон «с – по»);
   - тип программы (новичковая / опытная, множественный выбор);
   - конкретный шеф (множественный выбор);
   - переключатель «Только свободные» (`only_available = true`).
4. Система применяет фильтры согласно семантике: внутри группы действует логическое ИЛИ (если выбрано несколько типов программ или несколько шефов), между разными группами — И. Границы диапазона дат включительные, вводятся в локальной зоне клиента.
5. На экране отображается отфильтрованный список с сохранением сортировки по времени. Активные фильтры выделены, присутствует кнопка сброса.

### Альтернативные потоки

- **A1. Сброс фильтров:** клиент нажимает «Сбросить» — все фильтры очищаются, восстанавливается дефолтная выдача (7 дней, без фильтрации, `only_available = false`).
- **A2. Режим «Только свободные»:** активирован переключатель `only_available = true`. Заполненные и отменённые слоты полностью скрываются из списка.
- **A3. Семантика фильтров:** если выбраны, например, «Новичковая» и «Опытная» программы, показываются слоты обоих типов. При одновременном выборе нескольких шефов — классы любого из них. Между программами, датами и шефами действует логическое И.
- **A4. Пустой справочник шефов:** если по данным API нет ни одного активного шефа, фильтр «Шеф» автоматически скрывается, чтобы не загромождать интерфейс.

### Исключения / ошибки

- **E1. Нет слотов, соответствующих фильтрам:** отображается пустое состояние «Нет классов, подходящих под фильтры» и предложение изменить или сбросить критерии.
- **E2. Нет расписания вовсе:** в ближайшие 7 дней (или заданном диапазоне) нет ни одного предстоящего класса. Выводится empty state «Пока нет доступных классов» с пояснением, что расписание может появиться позже.
- **E3. Просмотр офлайн:** при отсутствии сети приложение показывает ранее загруженный кэш списка (если имеется) с явной пометкой «Данные могли устареть». Фильтры в офлайн-режиме не применяются, мутации запрещены. Доступна кнопка «Обновить» для повторного запроса при восстановлении связи.

### Постусловия

- Клиент видит актуальный (или помеченный как устаревший) список предстоящих классов. Данные расписания не изменяются.
