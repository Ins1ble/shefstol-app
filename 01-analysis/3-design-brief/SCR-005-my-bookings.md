## 1. Назначение и контекст

Экран даёт клиенту контроль над своими кулинарными классами — отдельно предстоящими и прошедшими. Это домашняя точка для проверки «куда и когда я записан», входа в детали записи и (через детали) запуска отмены. Экран — корневой во вкладке «Мои записи» таб-бара (см. `00-foundations.md` §4.2).

Контекст использования — кухня/студия, яркое освещение, спешка перед началом класса (`00-foundations.md` §1): клиенту нужно с одного взгляда понять ближайшую запись (когда начало, какая программа, сколько мест и какой инвентарь) и её статус.

---

## 2. Пользователь и цель

- **Роль:** Клиент (единственная роль приложения).
- **Цель пользователя:** быстро увидеть свои предстоящие классы, убедиться в деталях ближайшей записи, при необходимости перейти к деталям/отмене; при отсутствии записей — понять, как записаться.
- **Покрываемая история:** US-9 — «Как клиент, я хочу видеть список своих записей (предстоящих и прошедших), чтобы контролировать свои классы».

---

## 3. Навигация

### Входящие

- Таб **«Мои записи»** в таб-баре с любого корневого экрана АЗ (`00-foundations.md` §4.2).
- Из шторки BS-002 «Подтверждение записи» по действию «Перейти к моим записям» (сразу после успешной записи).

### Исходящие

- Тап по карточке записи → SCR-006 «Детали брони + отмена».
- Переключение вкладок таб-бара → SCR-002 «Список классов», SCR-007 «Профиль».
- Из Empty-состояния по действию «Записаться на класс» → переход к списку классов SCR-002.

### Каркас и таб-бар

Это корневой экран вкладки — таб-бар виден; на дочерних экранах (детали брони) скрывается по правилу `00-foundations.md` §4.2. Базовый каркас — `00-foundations.md` §4.1. Собственного нижнего CTA у экрана нет.

## 4. Контент-инвентарь

Данные — записи (брони) текущего клиента, см. модель данных. Числовые лимиты не зашиваются в UI — берутся из данных класса/программы (`00-foundations.md` §6).

### 4.1 Карточка записи (booking)

| Поле | Источник | Назначение на карточке |
|------|----------|------------------------|
| `status` | booking | Бейдж статуса: active → «активна», cancelled → «отменена», late_cancel → «поздняя отмена», cancelled_by_studio → «отменён студией». Текст + форма, не только цвет. |
| `start_at` | slot | Дата и время начала класса — крупно, основной ориентир. |
| `program.name` | slot → program | Название программы (меню) и её тип (если приходит). |
| `chef.name` | slot → chef | Имя шефа. |
| `price` / `price_total` | slot / booking | Цена (итоговая по записи). Оплата офлайн — текст см. `00-foundations.md` §6. |
| `seats_count` | booking | Число мест в записи (себя + гости). |
| Вариант инвентаря / `rental_count` | booking | Сводка по инвентарю: свой / прокатный (сколько прокатных наборов). Лейблы — `00-foundations.md` §6. |
| `dietary_restrictions` | booking | Если заполнено — иконка/краткий текст «Есть особые пожелания» (в предстоящих). В истории можно не показывать. |

> Карточка — сводка, достаточная для опознания записи; полная детализация — на SCR-006.

### 4.2 Разделение списка

- **Предстоящие** — только активные записи (status = active), у которых `start_at` в будущем (ориентир и приоритетная секция).
- **Прошедшие** — записи с `start_at` в прошлом И все отменённые (cancelled / late_cancel / cancelled_by_studio) независимо от даты начала. То есть отменённая запись с будущим началом тоже попадает в «Прошедшие».

> Правило отнесения: запись активна и начало в будущем → «Предстоящие»; иначе (начало в прошлом или запись отменена) → «Прошедшие». Активность и время начала вычисляются из данных, без отдельного флага. Прошедшие сортируются по убыванию даты начала (свежие сверху).

## 5. Структура и иерархия

Сверху вниз: хедер (заголовок + переключатель/секции) → скролл-список карточек → таб-бар.

```
┌──────────────────────────────┐
│  Мои записи                  │  ← заголовок экрана (хедер)
│  [ Предстоящие | Прошедшие ] │  ← сегмент-переключатель (или секции)
├──────────────────────────────┤
│ ┌──────────────────────────┐ │  ← скролл-список карточек
│ │ сб, 21 июн · 09:00       │ │     (карточка записи)
│ │ Итальянская паста        │ │
│ │ Новичковая · Шеф: Анна   │ │
│ │ 2 места · 1 прокатный    │ │
│ │ 3000 ₽          [активна]│ │  ← бейдж статуса (текст+форма)
│ └──────────────────────────┘ │
│ ┌──────────────────────────┐ │
│ │ вс, 22 июн · 18:00       │ │
│ │ Сложная выпечка          │ │
│ │ Опытная · Шеф: Игорь     │ │
│ │ 1 место · свой инвентарь │ │
│ │ 1500 ₽          [активна]│ │
│ └──────────────────────────┘ │
├──────────────────────────────┤
│  Классы · Мои записи · Профиль │  ← таб-бар
└──────────────────────────────┘
```

Группировка при варианте с секциями (вместо сегмента):

```
┌──────────────────────────────┐
│  Мои записи                  │
├──────────────────────────────┤
│  ПРЕДСТОЯЩИЕ                  │  ← заголовок секции (sticky)
│  [ карточка ]                │
│  [ карточка ]                │
│                              │
│  ПРОШЕДШИЕ                    │  ← заголовок секции (sticky)
│  [ карточка ]                │
│  [ карточка ]                │
└──────────────────────────────┘
```

Типографическая иерархия — `00-foundations.md` §3.3: дата/время начала и цена — наиболее контрастные элементы карточки.

## 6. Компоненты и поведение

| Компонент | Поведение |
|-----------|-----------|
| **Переключатель «Предстоящие / Прошедшие»** | Сегмент-контрол вверху или разделение списка на sticky-секции (выбор паттерна — за дизайнером). При сегменте показывается одна группа за раз; активная вкладка визуально выделена (текст + состояние, не только цвет). По умолчанию открыты Предстоящие. |
| **Карточка записи** | Кликабельна целиком (тач-зона ≥ `00-foundations.md` §3.1); тап → SCR-006. Крупная, в одну колонку, явно отделена отступами/границей. |
| **Бейдж статуса** | Текст + форма (контур/заливка/иконка), не только цвет — `00-foundations.md` §3.2, §3.5. Тексты — см. §8. |
| **Pull-to-refresh** | Жест «потянуть вниз» обновляет список (актуальные статусы/данные) — паттерн `00-foundations.md` §5. |
| **Скролл-список** | Вертикальный, одна колонка; при варианте секций — sticky-заголовки групп. |

## 7. Состояния экрана

Применяется сквозной паттерн `00-foundations.md` §5. Ниже — экранная специфика.

| Состояние | Содержимое / Поведение |
|-----------|------------------------|
| **Loading** | Скелетоны карточек записи (в форме будущего контента), без блокирующего спиннера. |
| **Content** | Список карточек, разделённый на Предстоящие (только активные) / Прошедшие (прошлые + все отменённые). |
| **Empty (нет записей вообще)** | Заглушка «У вас пока нет записей» + действие «Записаться на класс» → SCR-002. |
| **Empty в секции «Предстоящие» (есть только прошедшие)** | В секции/вкладке «Предстоящие» — короткая подсказка «Нет предстоящих записей» + «Записаться на класс»; секция «Прошедшие» при этом доступна. |
| **Error** | Заглушка ошибки нейтральным тоном + кнопка «Обновить». Текст сетевой ошибки — общий из `00-foundations.md` §6. |

## 8. Валидации и микрокопия

### Доступ к данным

- Клиент видит только свои записи (NFR-12, `00-foundations.md` §8.2). Чужих броней, контактов или административных данных на экране нет.

### Тексты бейджей статуса

| status | Текст бейджа |
|--------|--------------|
| active | «активна» |
| cancelled | «отменена» |
| late_cancel | «поздняя отмена» |
| cancelled_by_studio | «отменён студией» |

> Подробное пояснение поздней отмены (правило 2 часов, отсутствие штрафа) — на SCR-006 / BS-003; на карточке — только короткий бейдж.

### Empty-состояние

- Заголовок: «У вас пока нет записей».
- Подсказка/действие: «Запишитесь на класс» (ведёт на список классов).
- Тон — спокойный, без давления (`00-foundations.md` §6).

### Лейблы инвентаря и оплаты

- «Свой фартук и ножи» / «Прокатный набор», текст об офлайн-оплате — переиспользуются из `00-foundations.md` §6, не вводятся заново.

## 9. Доступность и mobile-first

Базовые правила — `00-foundations.md` §7 и §3. Экранная специфика:

- **Крупные карточки** с большой тач-зоной — удобно тапать мокрым пальцем на кухне.
- **Читаемые статусы:** бейдж дублирует смысл текстом и формой, считывается при ярком освещении и при дальтонизме.
- **Читаемые даты/время:** формат начала понятен с первого взгляда (день недели + дата + время), без двусмысленности.
- Переключатель/секции и карточки имеют доступные имена для screen reader; фокус-состояния обязательны.

## 10. Критерии приёмки дизайна

```gherkin
Сценарий: AC-001 Разделение на предстоящие и прошедшие
  Дано у клиента есть записи с разными датами начала и статусами
  Когда он открывает экран «Мои бронирования»
  Тогда записи разделены на группы «Предстоящие» и «Прошедшие»
  И по умолчанию отображаются предстоящие записи
  И в «Предстоящие» попадают только активные записи с будущим началом

Сценарий: AC-001a Отменённая запись с будущим началом — в «Прошедшие»
  Дано запись отменена (cancelled, late_cancel или cancelled_by_studio), но её начало ещё в будущем
  Когда клиент открывает экран
  Тогда эта запись показана в группе «Прошедшие», а не в «Предстоящие»
  И прошедшие записи отсортированы по убыванию даты начала

Сценарий: AC-002 Отображение статуса записи
  Дано запись имеет статус active, cancelled, late_cancel или cancelled_by_studio
  Когда клиент видит её карточку
  Тогда на карточке показан бейдж со статусом «активна», «отменена», «поздняя отмена» или «отменён студией»
  И статус передан текстом и формой, а не только цветом

Сценарий: AC-003 Пустой список
  Дано у клиента нет записей
  Когда он открывает экран
  Тогда показана заглушка «У вас пока нет записей»
  И доступно действие «Записаться на класс», ведущее к списку классов

Сценарий: AC-004 Только свои записи
  Дано в системе есть записи разных клиентов
  Когда клиент открывает «Мои бронирования»
  Тогда он видит исключительно собственные записи
  И на экране нет чужих или административных данных

Сценарий: AC-005 Переход к деталям брони
  Дано на экране показана карточка записи
  Когда клиент тапает по карточке
  Тогда открывается экран деталей брони (SCR-006)

Сценарий: AC-006 Обновление списка
  Дано экран в состоянии Content
  Когда клиент выполняет жест pull-to-refresh
  Тогда список перезапрашивается и отображает актуальные данные и статусы
```

## 11. Решения

| # | Решение |
|---|---------|
| 1 | **Отменённые записи — всегда в «Прошедшие»** (в т.ч. с будущим началом); «Предстоящие» — только активные (§4.2). |
| 2 | **Сортировка прошедших** — по убыванию даты начала (свежие сверху) (§4.2). |
| 3 | **Empty по секциям** — отдельная подсказка «Нет предстоящих записей» в секции «Предстоящие», если есть только прошедшие; общий empty «У вас пока нет записей» — когда нет ни одной (§7). |
| 4 | При отсутствии прошедших — секция/вкладка «Прошедшие» показывается пустой (не скрывается), чтобы навигация по секциям была предсказуемой. |
| 5 | *(Параметр конфигурации, не блокер)* Глубина истории прошедших — параметр данных/конфига; в MVP грузится доступная история целиком, лимит не хардкодим. |
