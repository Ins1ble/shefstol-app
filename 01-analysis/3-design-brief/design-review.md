## 2. Находки (с severity и привязкой к артефактам)

**Severity:** 🔴 Critical (нужно решение/правка до разработки) · 🟠 High · 🟡 Medium · ⚪ Low.

### 🔴 RR-D01. Лимит мест на бронь: «до 4» в дизайне vs «до 3» в ТЗ

- **Дизайн (Оформление записи, 124:6523):** подпись «Можно записать до 4 мест» при слоте со «Свободно мест: 5 из 8». То есть лимит = `free_seats`, без персонального потолка «себя + 2 гостя».
- **ТЗ/требования:** `max_seats = min(free_seats, program.capacity_cap, 3)` — групповой лимит «себя + до 2 гостей» (FR-12, feature-list §2, SCR-004, LOGIC-002).
- **Конфликт:** дизайн не применяет константу 3. Либо персональный лимит другой (4 = себя + 3 гостя), либо его нет вовсе (ограничивают только свободные места и потолок программы).
- **Затрагивает:** FR-12, SCR-004, feature-list §2, LOGIC-002, `slots/bookings` API.
- **Требует решения заказчика** (вопрос 1 ниже).

### 🔴 RR-D02. «Удаление аккаунта» отсутствует в макете профиля

- **Дизайн (Профиль, 124:6889):** только Имя, Телефон (инлайн-правка карандашами), «Правила студии», «Поддержка», «Версия приложения 1.0.0», кнопка «Выйти». Кнопки «Удалить аккаунт» **нет**.
- **ТЗ/требования/API:** SCR-007 описывает удаление аккаунта (FB-PROFILE-002), есть endpoint `deleteAccount` (DELETE `/profile`), последствия по FR-48 (аннулирование активных броней, анонимизация прошедших).
- **Конфликт:** функция есть в ТЗ и API, но не нарисована в макете.
- **Требует решения заказчика** (вопрос 2 ниже).

### 🟠 RR-D03. Карточка класса: hero-изображение программы

- **Дизайн (Карточка класса, 124:6456):** вверху карточки — фото-обложка программы (паста, выпечка и т.п.); ниже — название, тип, длительность, шеф, адрес, места, прокат, цена.
- **ТЗ/модель/API:** SCR-003 перечисляет параметры, но без обложки; сущность `Program` имеет `name`, `type`, `duration_min`, `description`, но нет явного поля изображения (`image_url`).
- **Действие:** добавить в SCR-003 элемент «обложка программы»; в модель данных и API добавить поле `Program.image_url` (опциональное, nullable). Если изображение не из MVP — отметить как декоративный плейсхолдер с фолбэком на цветной блок с иконкой программы.

### 🟠 RR-D04. Кнопка «Поделиться» на карточке класса

- **Дизайн (Карточка класса, 124:6456):** в хедере иконка «Поделиться» (Share Icon) рядом с «Назад».
- **ТЗ:** SCR-003 функцию «поделиться классом» не описывает.
- **Действие:** решить — специфицировать (share deep-link/текст класса с датой и программой) или отметить как декоративный/Phase 2 (вопрос 3 ниже).

### 🟡 RR-D05. OTP — фиксированные 4 ячейки; таймер повтора 00:30

- **Дизайн (SCR-001 шаг 2):** поле кода — 4 отдельные ячейки (segmented); таймер «Отправить код повторно (00:30)».
- **ТЗ:** длина кода — паттерн `^\d{4,6}$` (не зашивать в UI), таймер — из `resend_after_seconds`.
- **Действие:** в ТЗ SCR-001 и SCR-007 (смена телефона) указать UI-паттерн «segmented, число ячеек = длина кода из ответа API»; пример таймера привести к 30 с или оставить «из `resend_after_seconds`». Числа по-прежнему не хардкодить; уточнить у бэкенда фактическую длину кода (4 или 4–6).

### 🟡 RR-D06. Пищевые ограничения: чипы-теги vs текстовое поле

- **Дизайн (Оформление записи, 124:6523):** блок «Пищевые ограничения» — набор готовых чипов-тегов («Без глютена», «Без лактозы», «Вегетарианское», «Без орехов», «Аллергия») + свободное поле «Другое».
- **ТЗ:** SCR-004 описывает `dietary_restrictions` как **свободное текстовое поле** (FR-50).
- **Действие:** решить — оставить свободное текстовое поле (проще, MVP) или добавить набор пресетов-чипов (удобнее для клиента и шефа). При втором варианте: добавить в модель `dietary_tags: string[]` + текстовое поле `dietary_restrictions_other`. **Вопрос 4 ниже.**

### 🟡 RR-D07. Прокатные наборы: иконка «Фартук + ножи»

- **Дизайн (Оформление записи / Детали брони):** рядом с числом прокатных наборов — иконка «🧥 Фартук + ножи» как визуальный лейбл.
- **ТЗ:** SCR-004 и SCR-006 используют текстовые лейблы «Свой фартук и ножи» / «Прокатный набор» (foundations §6).
- **Действие:** синхронизировать микрокопию — принять «Фартук + ножи» как визуальный ярлык в дополнение к текстовому лейблу; обновить foundations §6 (добавить иконку как допустимое усиление, не замену текста).

### 🟡 RR-D08. Бейдж типа программы: цветной чип

- **Дизайн (Карточка класса / Список классов):** тип программы («Новичковая» / «Опытная») отображается как цветной чип (зелёный/оранжевый) с текстом внутри.
- **ТЗ:** SCR-002 и SCR-003 описывают тип программы как «текстовый бейдж», с требованием дублирования смысла формой/иконкой (foundations §3.2 — не только цветом).
- **Действие:** уточнить в ТЗ, что чип имеет **и** цвет, **и** иконку/форму, чтобы соответствовать NFR-1 (доступность при дальтонизме). Обновить SCR-002 и SCR-003 — добавить поле `program.type_icon` (или фиксированный набор иконок: 🟢🍳 новичковая, 🟠🔪 опытная).

### ⚪ RR-D09. Терминология заголовков

- **Дизайн:** заголовки/табы — «Классы», «Мои записи», «Профиль»; экран успеха «Вы записаны».
- **ТЗ:** внутренние названия «Список классов» (SCR-002), «Мои бронирования» (SCR-005).
- **Действие:** синхронизировать видимые подписи в ТЗ с дизайном (микрокопия), сохранив ID-экранов. Внутренние названия ТЗ оставить для трассировки.

### ⚪ RR-D10. Микрокопия пустых/ошибочных состояний

- **Дизайн:** Empty «Пока нет доступных классов. Загляните позже»; Error «Не удалось загрузить. Проверьте соединение и попробуйте снова. Обновить»; снек «Не удалось обновить данные. Проверьте интернет».
- **ТЗ:** каталог в [00-foundations §6] и LOGIC-008.
- **Действие:** сверить формулировки; расхождений не выявлено, но уточнить финальный копирайт перед передачей в разработку.

## 4. Решения заказчика (согласовано 2026-07-05)

### ✅ Вопрос 1. Лимит мест на одну бронь (RR-D01)

**Решение:** остаётся **3** («себя + 2 гостя»), формула `max_seats = min(free_seats, program.capacity_cap, 3)`.

- Подпись на макете «Можно записать до 4 мест» — **неточность дизайна**.
- ТЗ не меняем; передать дизайнеру правку макета — показывать динамический лимит `min(free_seats, cap, 3)`, а не захардкоженное «до 4».
- Обоснование: ограничение связано с физикой рабочего места и качеством обучения; больше 3 гостей на одного клиента — нагрузка на шефа.

### ✅ Вопрос 2. Удаление аккаунта (RR-D02)

**Решение:** остаётся **в MVP**.

- Отсутствие кнопки «Удалить аккаунт» в макете — **пробел дизайна**, не продукт-решение.
- Требование FR-48 и NFR-11 (GDPR/защита ПДн) обязывает функцию.
- ТЗ и API (`deleteAccount`) сохраняем; передать дизайнеру добавление кнопки на экран «Профиль» (ниже «Выйти», деструктивный стиль, менее акцентная).
- Подтверждение удаления — через мини-шторку (как описано в SCR-007).

### ✅ Вопрос 3. «Поделиться» классом (RR-D04)

**Решение:** **Phase 2 / декор** в MVP.

- В MVP не специфицируем; иконку на макете SCR-003 отметить как **неактивную** (disabled/hidden).
- В feature-list §7 «Не входит в MVP» добавить строку: «Поделиться классом (share deep-link) — Phase 2».
- Обоснование: основная ценность — запись на класс; виральность не является приоритетом старта.

### ✅ Вопрос 4. Пищевые ограничения: чипы vs текстовое поле (RR-D06)

**Решение:** **оставляем свободное текстовое поле** в MVP.

- Чипы-пресеты («Без глютена», «Без лактозы» и т.п.) — переносим в Phase 2.
- В MVP: одно многострочное текстовое поле `dietary_restrictions` (как описано в SCR-004).
- Обоснование: MVP должен закрывать базовый сценарий; структурирование аллергий требует дополнительного бэкенд-справочника и валидации. Шеф получит текст как есть.
- Передать дизайнеру: заменить набор чипов на одно текстовое поле с hint «Укажите аллергии и ограничения заранее».

### Остальные находки (RR-D03/05/07/08/09/10)

**Решение заказчика не требуется** — это приведение ТЗ к фактическому дизайну:

- **RR-D03 (hero-изображение):** добавить `Program.image_url` в модель и API (nullable, опциональное).
- **RR-D05 (OTP 4 ячейки / 00:30):** обновить SCR-001/007 — «segmented, число ячеек из API; таймер из `resend_after_seconds`».
- **RR-D07 (иконка «Фартук + ножи»):** обновить foundations §6 — иконка допустима как усиление, не замена текста.
- **RR-D08 (цветной чип типа):** уточнить SCR-002/003 — чип с **иконой + цветом + текстом** (доступность).
- **RR-D09/10 (микрокопия):** синхронизация копирайта перед разработкой.

## 6. Метод выгрузки (воспроизводимость)

- **Скрипт:** `GET /v1/files/{key}?depth=4` → сбор фреймов-экранов → `GET /v1/images` (PNG, scale 2) → скачивание + индекс.
- **Токен:** из `.env` (`FIGMA_TOKEN`, не коммитится в репозиторий).
- **Выгружено:** 63 фрейма (24 экрана + состояния + компоненты/цвета/типографика).
- **Хранение PNG:** вне репозитория (скретчпад `.figma-export/`, добавлен в `.gitignore`).
