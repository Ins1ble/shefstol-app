# Бриф для UI/UX дизайнера · «Шеф-стол»

Требования на дизайн клиентского мобильного приложения «Шеф-стол» (самостоятельная запись на кулинарные классы). Этот файл — обзор и индекс: единые правила вынесены в `00-foundations.md`, детальные требования — в отдельный документ на каждый экран (см. §«Экраны»).

## Цель и контекст

Приложение «Шеф-стол» заменяет ручную запись на кулинарные классы через WhatsApp и Google-таблицу, устраняя двойные брони, путаницу с местами и потерями продуктов. Клиент самостоятельно просматривает расписание, фильтрует, записывается (себя и гостей), выбирает инвентарь (свой или прокатный), отменяет записи, получает напоминания и может указать пищевые ограничения. Платформа — **нативное/гибридное мобильное приложение для iOS и Android** (mobile-first для использования на кухне и в студии), распространяется через App Store и Google Play (NFR-4). Веб-версии в скоупе нет.

**Скоуп — только роль «Клиент».** Шеф-повар и владелец работают через существующую инфраструктуру/админку и в приложение не входят. Справочные данные (классы, программы, шефы) — read-only из API; оплата — офлайн (наличные / перевод), приложение лишь показывает цену и фиксирует запись.

**Что входит в MVP (принятые решения):**
- **Профиль клиента (FR-47–FR-49):** просмотр и редактирование имени и телефона, выход из аккаунта, удаление аккаунта — экран SCR-007.
- **Цена и прокат (FR-45, FR-11–FR-14):** прокат инвентаря (фартук + ножи) — отдельный тариф. Итог записи = цена за место × число мест + тариф проката × число прокатных наборов. Своё оборудование бесплатно. Валюта — рубли (RUB). Итоговую цену считает и возвращает сервер; приложение показывает её, не пересчитывая.
- **Пищевые ограничения (FR-50):** необязательное текстовое поле при бронировании. Шеф получает информацию через существующую систему, клиент видит своё указание в деталях брони.
- **Адрес студии (FR-9a, NFR-26):** отображается текстом в карточке класса. При наличии координат может быть дополнен статической картой; при недоступности карты — текстовый fallback. Тап по адресу открывает шторку BS-004.
- **Отмена (FR-16–FR-18):** только целиком (нельзя отменить часть мест из записи). Правило 2 часов: ранняя отмена освобождает места и прокатные наборы, поздняя — фиксируется без штрафов.

Подробнее о продукте, аудитории, дизайн-принципах и контексте использования — `00-foundations.md` §1–2.

## Роли и их задачи

| Роль | Что делает в приложении |
|------|--------------------------|
| **Клиент** | Просматривает и фильтрует классы, записывается (себя + гости), выбирает свой или прокатный инвентарь, указывает пищевые ограничения, видит цену, отменяет записи, просматривает свои бронирования, получает напоминания. |
| Шеф / Владелец | **Не входят в приложение** — работают через существующую инфраструктуру/админку (вне скоупа). |

## Сквозные правила (foundations)

Единые для всех экранов требования вынесены в `00-foundations.md`: дизайн-принципы (mobile-first, минимальный порог входа, короткий путь к записи), структурные токены (контраст, тач-зоны, иерархия — без бренда), каркас и навигация (таб-бар, bottom sheet, нижний CTA), сквозной паттерн состояний (Loading / Content / Empty / Error), tone of voice и общая микрокопия, доступность, напоминания, безопасность данных, глоссарий.

**Объём визуальных требований.** Документы задают **функционально-структурные** требования (иерархия, компоненты, поведение, ограничения), без прескриптивных цветов/шрифтов — бренд в исходной аналитике не зафиксирован; визуальный язык определяет дизайнер.

## Пользовательские сценарии (user flows)

Ключевые пути навигации (полная диаграмма переходов будет детализирована в экранных документах):

- **Вход (трёхшаговый):** запуск → SCR-001 (телефон → код из SMS → имя для нового) → SCR-002.
- **Запись (UC-1):** SCR-002 → SCR-003 → SCR-004 → BS-002 → (SCR-005). ≤ 3 экранов до подтверждения (NFR-2). После первой записи в BS-002 запрашивается разрешение на push.
- **Фильтрация (UC-3):** SCR-002 → BS-001 → SCR-002 (с применёнными фильтрами; множественный выбор типа программы и шефа).
- **Отмена (UC-2):** SCR-005 → SCR-006 → BS-003 → SCR-006 (с обновлённым статусом). Отмена — только целиком.
- **Профиль:** SCR-007 — просмотр/редактирование имени и телефона, выход, удаление аккаунта; выход/удаление → SCR-001.
- **Адрес студии:** SCR-003 / SCR-006 → BS-004 (шторка с адресом и опциональной картой) → возврат на исходный экран.

## Состояния экранов

Единый паттерн для всех экранов с запросами — **Loading** (скелетон/шиммер) → **Content** → **Empty** (заглушка + подсказка) → **Error** (+ «Обновить»). Описан один раз в `00-foundations.md` §5; специфичные состояния (disabled CTA «Записаться» при 0 мест, бейдж «Поздняя отмена» и т.п.) — в соответствующих экранных документах.

## Тон, бренд, ограничения

- **Тон:** простой, прямой, дружелюбный, без давления и штрафов; обращение на «вы» — `00-foundations.md` §6.
- **Бренд:** не зафиксирован в аналитике — визуальный язык за дизайнером; обязательны лишь структурные ограничения (контраст, размеры, иерархия) — §3.
- **Доступность / mobile-first:** крупные тач-зоны, высокий контраст, читаемость на кухне и в студии — §3, §7.
- **Что НЕ делаем на старте (Phase 2+):** онлайн-оплата, оценки шефов, автоматическая отмена по форс-мажору, программа лояльности, публичные отзывы, интерфейсы шефа/владельца, SMS-уведомления.
